# Area report — Outsider Read-Through

> **Scope:** Roleplay: a non-engineer CS manager walks the front door and kickoff §0–1
> **Provenance:** One of seven independent Sonnet 5 (max effort) analysis agents from the Fable-orchestrated multi-lens review of 2026-07-06. Each agent read its files in full; findings marked as reproduced were verified in scratch repos or against live vendor docs by the agent itself. Evaluated *as if all ROADMAP items A–Y were built as designed* — reported gaps are beyond the roadmap.

## Summary

This area is the kit's outsider-facing front door: README.md (the pitch, philosophy, and file inventory), CHEATSHEET.md (a terse permission/sandbox mechanics reference), glossary.md (the kit's vocabulary), and Part 0-1 of claude-project-kickoff.md (one-time machine hardening plus the per-project setup ritual). Together these are what anyone — engineer or not — hits first when told "use this kit." The README does real conceptual teaching in plain prose (model+harness, directives/verifiers, a four-level safety ladder); CHEATSHEET and kickoff Part 0/1 are where the kit pivots from philosophy to literal JSON-editing and terminal mechanics aimed at people already running Claude Code. glossary.md sits between the two, part genuine term-definitions and part roadmap bookkeeping, and is reachable only by accident since nothing in README.md or claude-project-kickoff.md links to it.

## Strengths

- **The four-level safety pecking order (locked room / hard rules / agent's own judgment / plain-text instructions) teaches a real, transferable risk model in plain English with zero jargon prerequisite — a lay reader can hold and correctly apply it.**
  - *Evidence:* README.md lines 126-138: '1. A locked room, enforced by the operating system... This is the only true wall' through '4. Plain-text instructions... A sentence is never a safety guarantee.'
- **'Start skeptical' converts vague AI-unreliability unease into specific, sourced, checkable claims a non-engineer can repeat and verify.**
  - *Evidence:* README.md lines 39-52: 'Agents mark work "complete" without testing it'; agents 'confidently praise it — even when mediocre'; 'The safety classifier that reviews agent actions misses ~17% of genuinely overeager actions.'
- **Directives-vs-verifiers and the 'three readers, three documents' rule cleanly explain why the kit produces several files instead of one blob, without requiring prior harness vocabulary.**
  - *Evidence:* README.md lines 93-99 ('Directives point the agent... Verifiers check the agent's work...') and lines 278-281 ('The README is for people. The instruction file is for the agent... The wiki is for depth and history').
- **CHEATSHEET.md is self-aware about not being the newcomer's document and actively redirects a first-time reader elsewhere before the jargon starts.**
  - *Evidence:* CHEATSHEET.md lines 8-9: 'New here? Read the narrative field guide securing-claude-sessions.md first to build the mental model — this sheet is the terse reference behind it.'
- **The kickoff guide's Intake questions are written in genuinely plain, answerable language — an island of accessibility inside otherwise dense setup mechanics.**
  - *Evidence:* claude-project-kickoff.md line 157 (Q3): 'Name 2–3 files/dirs holding credentials or that must never be overwritten, even accidentally'; line 158 (Q4): 'The 5–6 shell commands you'll run daily.'
- **glossary.md's 'core frame' section gives clean, short definitions of the kit's own coined terms, each pointing to where it is elaborated further.**
  - *Evidence:* glossary.md lines 11-36 (Harness, Agent = Model + Harness, Directives & verifiers, The safety net, Builder & judge, The contract, The brief).

## Weaknesses

- **[HIGH]** Nothing in README.md, CHEATSHEET.md, glossary.md, or kickoff §0-1 ever states what 'Claude Code' literally is (a terminal/CLI tool distinct from the claude.ai chat interface most non-engineers know) or how to install/open it — the whole guided path assumes this prerequisite is already met. Severity is intention-conditional: high against intention 3 (a vibecoder who has only used chat-Claude has no way to even begin), low against intentions 1-2 (a kit for people already running Claude Code reasonably skips this).
  - *Evidence:* README.md's opening sentence: 'handed to a fresh Claude Code session at the very start of a new project' (lines 3-4) — no gloss. Grepping all four assigned files for 'terminal', 'command line', 'npm install', 'download', and 'what is Claude Code' returns zero explanatory hits; the term is used dozens of times and defined nowhere.
- **[MEDIUM]** CHEATSHEET.md's filename promises a beginner-friendly quick reference, but it is the densest, least-translated file among the four, and it is the only candidate location for terms like MCP, CLI, and CI, none of which are ever defined in any of the four assigned files.
  - *Evidence:* CHEATSHEET.md line 6 stacks five undefined terms in one clause ('The auto classifier is a probabilistic backstop. CLAUDE.md / autoMode prose is not a control'); line 18 uses 'Seatbelt on macOS / bubblewrap on Linux' with no gloss; line 27 says 'good for CI' with CI never expanded; MCP appears at lines 30 and 47 and is never defined in CHEATSHEET.md, README.md, or glossary.md (confirmed by grep of all three). A newcomer scanning an alphabetized root file list would plausibly open this file, whose name signals 'easy reference,' before or instead of README.md.
- **[MEDIUM]** glossary.md is not linked from README.md or claude-project-kickoff.md anywhere, so a reader following the kit's own stated read order never learns it exists; and its own claim about where security vocabulary lives is inaccurate — it says that vocabulary 'lives in ... CHEATSHEET.md,' but a full read of CHEATSHEET.md shows it contains zero term-by-term definitions, only terse usage.
  - *Evidence:* Grep for 'glossary' across README.md and claude-project-kickoff.md returns zero hits. glossary.md lines 106-110 ('Kept as-is'): 'the standard/borrowed vocabulary the kit speaks on purpose (deny/ask/allow, the sandbox, context rot, defense in depth, least privilege...) ... Security vocabulary lives in securing-claude-sessions.md and CHEATSHEET.md' — but CHEATSHEET.md (read in full for this review) never defines any of these terms, it only uses them. A second, separate '## Quick glossary' heading also exists in securing-claude-sessions.md (line 430, confirmed by grep, not read for this review) — two glossary-branded locations, neither linked from README, is its own fragmentation risk.
- **[MEDIUM]** Part 0's machine-hardening step places a root-owned file at a fixed system path with no acknowledgment that a company-managed laptop (MDM, no local admin) may block this, and the kit never signposts the softer, admin-free fallback it does actually support for that reader.
  - *Evidence:* claude-project-kickoff.md lines 62-63 and 79-82: 'Root-owned at /Library/Application Support/ClaudeCode/managed-settings.json ... hand-place it ... The real fallback exists but is buried: lines 296-299 note the per-repo 'sandbox.enabled' block is 'for a per-repo setup when you haven't done Part 0' — but nothing routes a no-admin reader there, and lines 447-450 name the specific unbypassable protections (disableBypassPermissionsMode, failIfUnavailable) as available only via the managed file, so a no-admin user silently loses exactly those guarantees without ever being told.
- **[MEDIUM]** The guide never explicitly states whether Part 0 is something the human does by hand or something Claude executes — the top-of-file instruction addressed to Claude says to run Part 1, and simply omits Part 0, leaving the reader to infer from context (root ownership) that this part is manual.
  - *Evidence:* claude-project-kickoff.md lines 10-13: '**How to use it (Claude, read this):** Execute Part 1 (setup) immediately... Internalize Part 2... Confirm setup is done, then ask for the spec.' Part 0 is never named in this instruction, and no other line in Part 0 or Part 1 states outright 'a human does this manually, outside any Claude Code session.'
- **[MEDIUM]** 'Auto mode' is used as load-bearing, assumed-known vocabulary throughout CHEATSHEET.md and the kickoff guide but is never given a first-principles plain-English definition (what it fundamentally is, versus only what it does or doesn't override).
  - *Evidence:* README.md never uses the phrase 'auto mode' or 'autoMode' at all (confirmed by grep) despite describing 'the agent works largely unattended' (line 258). CHEATSHEET.md's first uses (lines 6, 13, 16) assume prior knowledge. claude-project-kickoff.md line 119 ('Auto mode is the assumed default at every tier') and line 1120 ('enter auto mode (Shift+Tab cycles the permission mode)') both use it relationally rather than defining it as, e.g., 'the setting where Claude executes actions without pausing to ask permission each time.'
- **[MEDIUM]** The setup ritual assumes the reader already has 'a stack' — a language, package manager, test runner, and linter — with no branch for someone whose prior AI-built project lived entirely inside a chat interface or a no-code/low-code builder with no visible file tree, git repo, or terminal.

## Fit against the kit's three intentions

### As a kickoff point for new projects

Reasonably strong for a reader who already has Claude Code running and baseline terminal/git literacy: the README's philosophy is clear, the kickoff guide's JSON blocks are copy-paste-ready, and the Intake questions (§1.0a) front-load the right decisions in plain language. It misses even for a semi-technical starter in two ways found in this area: Part 0 conflates "install once per machine" with a procedure only someone with admin rights can complete, without saying so; and the guide's own framing ("How to use it (Claude, read this)") assumes a human hands the whole file to an agent rather than reading it themselves first, so a human who does want to skim before delegating has no equivalent human-paced walkthrough — they either read agent-directed prose or nothing.

### As an adoption skill for existing projects

I did not read claude-project-adoption.md directly (outside this area's assigned scope), so this is reasoned from README.md's framing plus the onboarding-prerequisite logic that plainly carries over. README.md (lines 154-160) pitches adoption in one clear, plain-language paragraph: "a thorough read-only evaluation, an interview... a written proposal the owner approves before any edit." That description itself lands fine for a non-engineer. But every prerequisite gap this review found in the shared front door — Claude Code never explained, Part 0's admin assumption, CHEATSHEET's jargon density, an unlinked glossary — sits upstream of claude-project-adoption.md and applies at least as much to it, since retrofitting an existing project plausibly skews toward owners even less likely to already be Claude Code power users than someone deliberately starting an AI-native project. Whether claude-project-adoption.md itself compensates for any of this is outside what I read.

### As a teaching tool beyond vibecoding

Serves intention 3 unevenly, and roughly backwards from what a "beyond vibecoding" reader needs first: strong on philosophy, absent on mechanics. Read in character as a customer-success manager who has used Claude/ChatGPT for a year, built one small vibecoded tool, and never used git seriously or written bash: the README's conceptual sections genuinely land — the model+harness framing, the four-level safety ladder, and "start skeptical" each taught something true and memorable enough to repeat to a boss without embarrassment. The moment the kit turns operational, though, there is no on-ramp. "Install the kit as a skill... clone or symlink this repo to `~/.claude/skills/harness-kit`" (README.md lines 240-241) uses three unexplained operations in one clause — I could not perform this step. CHEATSHEET.md's very first substantive line packs five undefined terms into one sentence (line 6), and MCP/CLI/CI are used throughout the kit without ever being spelled out in any of the four files I read. Nothing tells me Claude Code is a distinct terminal application from the chat interface I already know, or how to get it — the single most basic prerequisite is assumed, never taught. Part 0 asks me to place a file at a root-owned system path; if my laptop is company-managed I don't know whether I even can, and the kit doesn't say. glossary.md — the one document built to solve exactly my jargon problem — is never linked from anywhere I would organically read, and even if I stumbled onto it via the raw file list, roughly a quarter of it is a roadmap ledger I have no context for, and its own scoping note explicitly declines to define "the sandbox," "least privilege," and "defense in depth" as vocabulary I'm assumed to already have. The kit teaches me to think like a skeptical harness-builder; it does not teach me how to physically become one starting from zero.

## Understand deeply vs. trust Claude

Must understand deeply: that a safety boundary's strength equals the agent's inability to reach it (README's four-level pecking order) — this is the one idea that actually protects a newcomer from real harm (a leaked secret, a destructive command), and no amount of trusting Claude substitutes for a human knowing which of the four levels a given rule actually lives at. Equally load-bearing: that the durable harness lives in project files, never in a chat conversation or the agent's private memory (README's "knowledge in the project, not the agent's memory") — a newcomer's instinct after a year of chat-based use is to treat spoken instructions as durable, and this kit's entire safety model depends on unlearning that instinct specifically. What can safely be trusted to Claude: the exact JSON syntax of a settings file, the specific glob/regex patterns in a deny list, the precise wording of a CLAUDE.md skeleton, and the terse, version-pinned mechanical facts in CHEATSHEET.md — provided the reader later re-verifies per the kit's own "prove it bites, don't trust a self-report" discipline (kickoff §1.4). A newcomer doesn't need to memorize `sandbox.network.allowedDomains` syntax, but does need to know why an allowlist can never un-leak a secret once it has been read.

## Durability over the next 1–2 years

The mechanical layer this area leans on hardest — CHEATSHEET.md's exact key names, precedence quirks, and "verified against Claude Code 2.1.x docs" facts — is precisely what the kit's own permanent/depreciating/appreciating framework (README, "What scales with the model, and what doesn't") predicts will rot fastest: it is tied to one product's current settings schema, and the document flags its own volatility by version-pinning itself. If Claude Code's onboarding matures (a setup wizard, a GUI settings editor, an installer that handles Part 0 automatically), today's terse JSON-editing and root-file-placement instructions could become unnecessary quickly. What stays vital regardless of model or product improvement: the conceptual safety ladder itself (a control's strength is about reachability, not about how smart the model is — a smarter model doesn't change who is allowed to edit a settings file), and, specific to this area, the raw fact that a first-time non-engineer reader needs a plain-English front door before any mechanics at all. That is a documentation/product-design problem, not a capability problem — a smarter Claude embedded in a smarter harness does not retroactively write the "what is this tool and how do I open it" page for the reader who hasn't started yet.

## Missing beyond the roadmap (A–Y)

- A genuine 'absolute zero' on-ramp: what Claude Code is as distinct from claude.ai, how to install it, and minimum terminal/shell orientation (open a terminal, what a prompt is, what typing a command does) — no roadmap item (A-Y) addresses the reader's very first action.
- A newcomer-facing glossary actually linked from README.md that defines the terms a non-engineer needs (MCP, CLI, CI, and — despite glossary.md's claim they live elsewhere — a working definition of 'the sandbox', 'least privilege', 'defense in depth' reachable from the vocabulary doc itself) rather than assuming them as already-known 'standard vocabulary'.
- Guidance for a no-admin / company-managed (MDM) laptop on exactly which Part 0 protections are and aren't available, and how to proceed safely without the managed floor — the kit has the fallback (per-repo sandbox) but never signposts it for this reader.
- Any accommodation for a reader whose prior AI-built project has no git repo, no package manager, and no terminal history at all (built via a chat interface or a no-code tool) — Part 1's Intake Q1 and the stack-agnostic disclaimer both presuppose a traditional codebase already exists or is confidently being chosen.
- A single, sequenced reading path for a first-time non-engineer across the kit's now-large file set (14+ root docs) — the 'What's in here' section in README is an inventory, not an ordered path, and the kit's own recommended entry point for 'a human leveling up' (LESSONS.md, per README line 212) sits in the same directory as CHEATSHEET.md, whose name reads as far more inviting to a first-time browser despite being far less appropriate for a first read.

## Exceptional

- The four-level safety pecking order in README.md (locked room / hard rules / agent's own judgment / plain-text instructions) — exceptional plain-English risk communication that requires zero prior jargon and gives a lay reader a mental model they could correctly apply to a tool they have never used.
- The 'start skeptical' section's grounded, numbered, sourced failure-mode claims — it converts vague 'AI can be unreliable' unease into specific, memorable, checkable facts, unusually disciplined technical communication for a section that could easily have been generic hand-wringing.

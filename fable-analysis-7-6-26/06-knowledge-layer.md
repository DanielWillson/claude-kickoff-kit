# Area report — Knowledge & Memory Layer

> **Scope:** llm-wiki-kickoff.md · wiki/ (all pages) · LESSONS.md
> **Provenance:** One of seven independent Sonnet 5 (max effort) analysis agents from the Fable-orchestrated multi-lens review of 2026-07-06. Each agent read its files in full; findings marked as reproduced were verified in scratch repos or against live vendor docs by the agent itself. Evaluated *as if all ROADMAP items A–Y were built as designed* — reported gaps are beyond the roadmap.

## Summary

The knowledge/memory layer is llm-wiki-kickoff.md (704 lines) — a from-scratch methodology for a Karpathy-style "LLM wiki," an agent-maintained, git-versioned Markdown knowledge base that reconciles its claims against ground truth (source code, or a `verified:` date for non-code claims) instead of against its own internal consistency. LESSONS.md (191 lines) distills seven field-tested epistemic lessons (verification-is-the-bottleneck, context-as-budget, harness shelf-life, "instructions are dependencies," etc.) that the kit's operational guides apply but don't re-derive. The kit dogfoods the pattern on itself: wiki/ holds three source-verification pages (Anthropic writing, operator field reports, a Claude Code feature matrix), five ADR-lite decision records, and an append-only harness-change journal (wiki/harness-log.md, 589 lines) recording the kit's own last three days of evolution as dated bets with open retrospects. Together these are the kit's "why" layer — the citations and decision history everything else leans on without restating inline.

## Weaknesses

- **[HIGH]** The wiki's central mechanism — the maintenance engine that makes the pattern self-correcting rather than another rotting docs folder — ships nowhere as runnable code, unlike every sibling verifier the kit ships, and the audit doesn't even detect the gap. This means the 'trust banners' and `verified:` dates the pattern relies on are, as shipped, an honor system.
  - *Evidence:* No lint/reconcile/stale/coverage/gaps/metrics implementation exists anywhere in the repo (confirmed: only claude-audit-base.sh, claude-eval-base.sh, scripts/harness-metrics.sh, scripts/kit-conformance.sh exist; none implements the six subcommands llm-wiki-kickoff.md:354-449 specifies in operational detail). ROADMAP.md:62-67 states the kit's own standard — each practice lands as 'the teaching + a starter template/script' in the kit; audit/eval/metrics/conformance all ship a runnable `.sh` starter, but the wiki ships only page-template shells (llm-wiki-kickoff.md §8, ~576-686), not an engine script, in the one practice where the engine is hardest to write (YAML parsing + wikilink-graph resolution + git-diff mapping) and most load-bearing. claude-audit-base.sh:35-37 ships an empty `WIKI_LINT_CMD=""`, and :677-685 prints the identical neutral '· no wiki lint configured' bullet whether or not `wiki/` exists — no WARN distinguishes 'no wiki' from 'wiki exists, engine never built.' scripts/kit-conformance.sh:190-198 (item O) only counts incident/decision-shaped page presence (`WIKI_MIN=3`), not engine existence. Checked ROADMAP A–Y: item D targets graduating past markdown at multi-project scale (a different problem), and O is presence-only by explicit design — none proposes shipping or checking for this engine.
- **[MEDIUM]** The harness-log's own reason to exist — 'closing the loop' on whether a bet paid off — has no staleness/reminder machinery forcing a revisit, the same category of silent-aging gap the wiki pattern solved for no-`code:` pages via the `verified:` clock, but never turned on its own artifact.
  - *Evidence:* wiki/harness-log.md:34-35 ('Close the loop... an unrevisited log is just a changelog'); every one of the six entries carries an open 'Retrospect' with no analog to the `stale`/`verified:`-clock mechanism several entries reference by name (e.g. :229 'mirroring the wiki's stale/coverage') turned inward on the log's own bets.
- **[LOW]** A schema/instance drift inside the one document most focused on schema-vs-enforcement drift elsewhere: every entry carries an undeclared, inconsistently-named section not in the file's own stated 9-field schema.
  - *Evidence:* wiki/harness-log.md:37-39 declares the schema as 'date · change · rationale (the bet) · what it replaced · shelf-life/risk class · related ROADMAP item · commit · signal to watch · Retrospect'; every entry also carries either a 'Design choices worth pointing at' block (lines 90, 163, 247, 402, 465) or a 'Code worth pointing at' block (lines 516, 568) — two undeclared variants, appearing in 100% of entries, neither named in the schema.
- **[LOW]** A minor paraphrase-drift in exactly the failure mode the corpus's own Lesson 7 warns against: presenting a diagnosis figure as if it were also a measured result of the fix.
  - *Evidence:* wiki/sources/operator-field-reports.md:20-22 says consolidating docs 'cut processing time ~40% and tokens ~75%'; my direct fetch of the source found the ~40% figure describes time lost to doc confusion before the fix (the diagnosis), with only ~75% reported as an after-the-fix measured result. LESSONS.md's own numbers table (line 180) states this more precisely ('agent time lost to contradictory docs / token cut from consolidating').
- **[LOW]** The pattern's single foundational citation — Karpathy's gist, which everything in llm-wiki-kickoff.md operationalizes — gets no `verified:`-dated wiki/sources/ page of its own, unlike the Anthropic and operator sources; its only URL+quote citation lives in a different file (README.md) from the guide it grounds.
  - *Evidence:* llm-wiki-kickoff.md:17 names 'Karpathy's pattern, April 2026' with no URL; README.md:370-380 carries the actual citation, gist URL, and verbatim quote (confirmed to resolve and match); wiki/index.md's Sources list (:8-16) covers Anthropic/operator-reports/feature-matrix but has no Karpathy entry.

## Fit against the kit's three intentions

### As a kickoff point for new projects

Conceptually strong onboarding — §1 is unusually honest about when not to bother ("don't bother for a throwaway task," llm-wiki-kickoff.md:27) — but the fit is weaker here than for adoption. A brand-new project has no git history to seed the highest-value pages from: §5's "seed 2-3 real incident/decision pages from genuine history" (:470-471) has nothing to draw on at hour zero, and §6's own closing line — "critical mass is real... don't judge it by the first run" (:548-549) — concedes the value is deferred. That deferral compounds with the missing engine (see weaknesses): the adopter most likely to attempt a wiki at kickoff (a solo hobbyist) is also least likely to have the bandwidth to hand-build six subcommands of YAML/wikilink/git-diff tooling from a prose spec with zero starter code. In practice a fresh-kickoff wiki plausibly stalls at "wrote SCHEMA.md and two pages, never built reconcile" — precisely the silently-stale outcome §1 calls worse than no wiki at all.

### As an adoption skill for existing projects

Stronger fit than kickoff. An existing project supplies exactly what a fresh one lacks — real git history, real incidents, real contradictory docs to consolidate — and §5-6's migration guidance reads as specific, hard-won operational wisdom rather than generic advice: the breadcrumb-migration pitfall naming exact file extensions a real pass missed (llm-wiki-kickoff.md:481-487), and "a hand-maintained status doc is the canonical rot — derive status from tests + git log instead" (:504-509). This is also where the Gustafson citation (independently verified) lands hardest: an agent losing ~40% of its time to contradictory existing docs is exactly the adoption-time pain this guide targets. The same engine-gap weakness still applies once migration is done, but the migration-specific counsel is a genuine, adoption-unique strength the kickoff case doesn't get to use.

### As a teaching tool beyond vibecoding

LESSONS.md is well-pitched for the stated audience — someone technically literate stepping past vibecoding — because it teaches epistemics (why convergent evidence across four independent sources matters, Lesson 1; why self-grading is mechanically biased, not just occasionally sloppy, Lesson 2) rather than bare rules, and models the target behavior by confessing its own near-miss (Lesson 7's retraction of an unsourced "50-line rule," reinforced by a decision record I independently verified). llm-wiki-kickoff.md is the denser, builder-facing document by comparison — §2.9's shingle-linter/corpus-scan discussion, §2.10's register mechanics, the epistemic-class taxonomy §2.1a explicitly declines — appropriate for someone about to build the pattern, but it presumes fluency in concepts LESSONS.md is itself busy teaching. The kit's own stated sequencing (LESSONS.md:6-8: "the kit's other documents apply these lessons; this one explains them") gets this ordering right; the risk is only that a reader who opens llm-wiki-kickoff.md first, unguided, will likely bounce off its density.

## Understand deeply vs. trust Claude

Must understand DEEPLY: (1) why reconciling against ground truth, not self-consistency, is the only fix for a confidently-stated fact gone silently stale — generic "agent memory" has no oracle to check against; a codebase or a dated human confirmation does (§2.1/§2.1a). (2) Why memory-layer placement is decided by when a fact loads and whether it has a truth-loop, not by format — otherwise a builder reflexively dumps project knowledge into machine-local memory that then pollutes other projects (§2.2); I confirmed live that Claude Code's Auto Memory is now genuinely wiki-shaped (an index file plus cross-linked topic files), which makes this transport/ground-truth distinction the only thing still doing real work. (3) Why a fresh, separate judge context is structurally necessary, not just good practice — errors within one context are correlated (LESSONS.md Lesson 2). (4) The honest limits of every automated check in this corpus — the link-graph drift detector (§2.9) only catches linked pairs; nothing catches two pages that never link and silently diverge — so a builder doesn't over-trust green output. The deepest and least obvious skill is (5) scale-calibration: knowing when NOT to build the machinery, which the kit's own wiki models (wiki/SCHEMA.md:19-24) but which an eager adopter easily gets backwards, over-building a six-subcommand engine for what should stay a five-page personal wiki. Can safely TRUST Claude wrote well: the mechanical execution (YAML frontmatter shape, marker syntax, decision-record prose) and the *initial* citation capture — I independently verified five citations against live primary sources and all five matched. What can't be outsourced even here: owning the ongoing re-verification cadence — claude-code-feature-matrix.md says outright it is "exactly the kind of page that rots" (line 16), and nothing in the kit schedules its own re-check.

## Durability over the next 1–2 years

Likely to need refresh or become unnecessary: the specific numeric snapshots (feature-matrix version gates, install counts, the CLAUDE.md line-count target itself) are model/tool-generation facts — the kit's own "depreciating" category — though the discipline of dating and self-labeling them as perishable is durable precisely because it doesn't depend on the facts staying true. More structurally: if Claude Code's Auto Memory (confirmed live to already be an index file plus small cross-linked topic files) grows a project-scoped, git-tracked, reconcile-checked variant, much of the hand-rolled engine spec in llm-wiki-kickoff.md §4 could be replaced by configuring a native feature instead of writing one from a 704-line prose spec — which would also close this area's biggest weakness for free. Stays vital regardless of model/harness improvement: reconcile-against-ground-truth as a category, because a more capable, more autonomous agent has more opportunity to act confidently on a stale belief for longer before a human notices (the kit's own "appreciating" framing; the "Confident and Wrong" paper's 68-80%-invisible-failure finding cited in ROADMAP.md §5) — this problem gets worse, not better, as models improve. Likewise one-claim-one-home, the tensions register for human-only-adjudicable conflicts, and builder/judge separation are architecture-level and model-agnostic — a smarter model doesn't remove the correlated-error problem within one context. And the context-rot-driven case for a small, link-navigated corpus over RAG (§7) is durable because rot is about attention allocation degrading with volume, not a capability ceiling a bigger window alone fixes.

## Missing beyond the roadmap (A–Y)

- No ROADMAP item (A-Y) proposes shipping or requiring a reference implementation of the wiki's own maintenance engine (lint/reconcile/stale/coverage/gaps/metrics) — the most operationally detailed piece of machinery anywhere in the kit (llm-wiki-kickoff.md §4), yet the kit ships zero code for it. Checked: D targets graduating past markdown at multi-project scale, a different problem; O's conformance check is presence-only by explicit design.
- No item makes the audit (or O) detect 'a wiki/ directory exists but its lint command was never wired' — the one condition that would catch an adopter who followed the guide's page-format advice but never built the freshness engine, leaving exactly the confidently-stale, unflagged wiki §1 calls worse than no wiki.
- No item adds a staleness/reminder check for the harness-log's own open 'Retrospect' fields — the log needs the same fix the wiki pattern already invented for itself (the `verified:` clock for no-`code:` claims) but never turns inward.
- The wiki's `verified:`-before-a-major-Claude-Code-release trust banner (wiki/index.md:5-6, wiki/SCHEMA.md:14-15) names a precise, in-principle machine-checkable trigger — Claude Code already exposes a version string cited throughout claude-code-feature-matrix.md — but no item wires a check against it; the banner is currently pure honor system.

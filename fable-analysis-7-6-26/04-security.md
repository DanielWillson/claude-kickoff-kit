# Area report — Security Surface

> **Scope:** securing-claude-sessions.md · templates/ · .claude/settings.json · claude-audit-base.sh
> **Provenance:** One of seven independent Sonnet 5 (max effort) analysis agents from the Fable-orchestrated multi-lens review of 2026-07-06. Each agent read its files in full; findings marked as reproduced were verified in scratch repos or against live vendor docs by the agent itself. Evaluated *as if all ROADMAP items A–Y were built as designed* — reported gaps are beyond the roadmap.

## Summary

The kit's security surface is a five/seven-level permission model — conversation mode/classifier → committed project settings → OS sandbox → root-owned managed settings → server-side (GitHub) — taught narratively in securing-claude-sessions.md, compressed into a terse verified-mechanics sheet in CHEATSHEET.md, and instantiated in three JSON templates (managed-settings.template.json, project.settings.json, project.settings.local.json.example) plus the kit's own dogfooded .claude/settings.json. claude-audit-base.sh adds the detection layer: secret/entropy scanning, dependency-CVE scanning, and a self-verifying guarded() helper that stops rot-prone greps from silently reading as "pass" once their target is renamed away. The organizing doctrine — "a control is only as strong as the agent's inability to reach it" — is used consistently to separate soft, agent-editable convenience rules from hard, admin-only guarantees. I verified the core technical claims (deny-rule/Bash interaction, sandbox default read/write asymmetry, hostname-only proxy without TLS inspection, JSON schema keys) live against Anthropic's current (2026-07) Claude Code docs and found them accurate, and independently re-verified the guarded() mechanism by extracting it into a standalone test harness and running all three documented cases.

## Strengths

- **The deny-rule/Bash-bypass analysis is not just correct but current-to-the-week, verified against live Claude Code docs I fetched during this review. CHEATSHEET.md's claim that 'A Read/Edit deny binds native file tools AND recognized Bash readers (cat/head/tail/sed/grep) — but NOT arbitrary code reading a file (python3 -c open(), php -r)' matches Anthropic's docs almost verbatim: 'Read and Edit deny rules apply to Claude's built-in file tools and to file commands Claude Code recognizes in Bash, such as cat, head, tail, and sed. They don't apply to arbitrary subprocesses that read or write files indirectly, like a Python or Node script that opens files itself.' Getting this exactly right, including the specific bypass class, is the single most consequential fact in the whole model, and most teams get it wrong or never learn it.**
  - *Evidence:* CHEATSHEET.md:36 vs. code.claude.com/docs/en/permissions ('Read and Edit' warning box), fetched live 2026-07-06
- **The sandbox's network-egress limits are described with real precision, not marketing language: hostname-only proxy decisions with no TLS inspection, and an explicit domain-fronting caveat. This exactly matches Anthropic's current 'Security limitations' section, down to naming domain fronting as the specific bypass.**
  - *Evidence:* securing-claude-sessions.md:212-214 ('a proxy that decides from the requested hostname without inspecting TLS... a determined exfil can use domain-fronting') vs. code.claude.com/docs/en/sandboxing 'Security limitations': 'the proxy makes its allow decision from the client-supplied hostname without inspecting TLS, code running inside the sandbox can potentially use domain fronting...'
- **The guarded() safeguard-rot mechanism in claude-audit-base.sh is sound bash and works exactly as documented. I extracted it into a standalone harness and ran the three claimed cases (anchor present, anchor file renamed, anchor symbol removed) independently; all three produced the documented behavior with no false pass.**
  - *Evidence:* claude-audit-base.sh:88-105; independently reproduced at /private/tmp/.../scratchpad/guarded_fixture — output: 'CASE 1... PASS', 'CASE 2 (renamed)... WARN, body_ran=0', 'CASE 3 (symbol gone)... WARN, body_ran2=0'
- **The kit never oversells a control's strength — every deny-rule template carries an inline caveat about its own limits (e.g. Bash catastrophic-command denies are explicitly labeled 'a BACKSTOP: deletion is bypassable, so the sandbox is the real confinement'), which is unusual honesty for a security scaffold and matches how the actual mechanism behaves.**
  - *Evidence:* templates/managed-settings.template.json:20; project.settings.json:32-34 ('a Bash redirect... is an in-project write the sandbox allows, so in-repo guard hooks are BACKSTOPS, not boundaries')
- **A concrete, non-generic worked example shows real defense-in-depth reasoning (7 independent controls across 4 levels protecting one credential) rather than asserting the phrase.**
  - *Evidence:* securing-claude-sessions.md:330-352
- **Correct understanding of a genuinely subtle mechanic: permission arrays (deny/ask/allow/allowedDomains/excludedCommands) merge across scopes rather than override, which is what lets a managed deny be authoritative without needing the heavier allowManaged*Only enterprise lockdown — confirmed against the docs' settings-precedence and sandbox-merge sections.**
  - *Evidence:* templates/README.md:3-6 vs. code.claude.com/docs/en/sandboxing ('When the same filesystem array is defined in multiple settings scopes, the arrays are merged')

## Weaknesses

- **[HIGH]** The kit's only hard guarantees (yolo disabled, forced sandbox, machine-credential denies, catastrophic-Bash denies) live exclusively in the managed-settings floor, which is a manual, root-privileged, per-machine, hand-placed file with zero installer automation and zero durable verification — and this is true even after every ROADMAP item (A-Y) ships, because item O's own conformance script deliberately treats it as a permanent, silent SKIP rather than a check. There is no mechanism anywhere in the kit — at kickoff, at adoption, or ongoing — that will ever tell a user 'your hard floor is not installed.' Given Claude Code's own docs confirm that, out of the box, plain file reads (including ~/.ssh/id_rsa) require no approval and the sandbox's default read policy still allows ~/.aws/credentials and ~/.ssh, the gap between 'fully protected per this guide' and 'stock zero-protection default' is exactly one skipped, un-nagged, un-automated step.
  - *Evidence:* ROADMAP.md:201-202 ('The managed floor is a loud SKIP... SKIPPED ≠ PASS'); templates/README.md:34-37 ('Hand-place it, then run claude doctor... and /status'); no installer script found anywhere in the repo; confirmed against code.claude.com/docs/en/sandboxing ('this default still allows reading credential files such as ~/.aws/credentials and ~/.ssh/') and code.claude.com/docs/en/permissions ('Read-only | File reads, Grep | Approval required: No')
- **[HIGH]** The template the kit actually hands to new projects ships weaker credential protection than the kit gives itself, and its own comment names the exact assumption that the hole above violates. templates/project.settings.json's deny list covers only in-project paths (./.env, ./secrets/**) with the comment 'machine creds are already denied in the managed floor' — but the kit's own dogfooded .claude/settings.json adds a redundant repo-level deny for ~/.ssh/**, ~/.aws/**, ~/.npmrc, *.pem, *.token anyway, as exactly the kind of belt-and-suspenders duplication the guide elsewhere argues for ('the most critical deny rules... are also written into the managed file... so they hold even if the repo copy were tampered with'). The template doesn't backport its own maintainer's practice into what it ships to others; this is a cheap, concrete, two-line fix that would materially shrink the blast radius of weakness #1.
  - *Evidence:* templates/project.settings.json:28-31 vs. .claude/settings.json:14-17; securing-claude-sessions.md:187-188
- **[MEDIUM]** No template or starter example anywhere gates MCP servers or WebFetch domains by default, despite prose in three separate files correctly identifying MCP and WebFetch as unsandboxed, un-audited surfaces equivalent in risk to Bash. A repo-neutral 'deny mcp__* by default, allowlist per server' line would be cheap and consistent with the kit's own Bash-deny pattern, but doesn't exist.
  - *Evidence:* claude-project-kickoff.md:502-506 ('MCP servers and the native web tools are unsandboxed, un-audited trusted code... allowlist exactly the servers you trust') — but grep for 'WebFetch(domain' or 'mcp__*' across templates/*.json returns zero template lines, only prose
- **[MEDIUM]** claude-audit-base.sh's own comment argues CI enforcement of the audit is strictly better than the client-side pre-commit hook it also ships wiring for, yet the kit contains no CI workflow template anywhere to realize that recommendation — a gap against the kit's own stated 'kit vs repo' rule that every practice ships a starter template, not just prose.
  - *Evidence:* claude-audit-base.sh:304-307 ('running this whole script as a CI step is the TOOL-AGNOSTIC enforcer... a client-side pre-commit hook never sees [this], which does NOT run in CI'); no .yml/.yaml file exists anywhere in the repo (verified by find)
- **[MEDIUM]** managed-settings.template.json excludes docker * from the sandbox entirely (following Anthropic's own documented guidance, since Docker is itself sandbox-incompatible), but ships this as an unqualified wildcard alongside gh/gcloud/terraform/brew with no companion ask/deny on privilege-escalating flags. A single docker run --privileged or a docker-socket bind-mount is a well-known one-command host-root escape — Anthropic's own sandboxing docs flag this exact risk under 'Privilege escalation via Unix sockets' — and it sits in a default-approved carve-out that reads, in the template, as no riskier than its list-mates.
  - *Evidence:* templates/managed-settings.template.json:37; code.claude.com/docs/en/sandboxing 'Security limitations': 'allowing access to /var/run/docker.sock effectively grants access to the host system through the Docker socket'
- **[LOW]** The auto mode the kit adopts as its daily-driver default is documented by Anthropic as 'Currently a research preview,' but securing-claude-sessions.md presents it as simply 'the default mode... a smart filter' without flagging that its behavior is explicitly unstable/beta — a gap in a document whose entire thesis is knowing exactly how strong each layer is.
  - *Evidence:* securing-claude-sessions.md:54 ('On our machines, auto is the default mode'); code.claude.com/docs/en/permissions permission-modes table: 'auto | Auto-approves tool calls with background safety checks... Currently a research preview'
- **[LOW]** The Stop-hook auto-commit convenience feature stages and commits ALL tracked changes, including to .claude/settings.json itself, under a routine 'WIP: auto-saved' message on every session end, with no integrity check anywhere in claude-audit-base.sh comparing settings.json against a known-good baseline — so a settings file altered mid-session (via a mechanism the docs describe as sandbox-dependent, not independently verified here) could be folded into history as unremarkable noise unless a human specifically scrutinizes that diff.
  - *Evidence:* templates/project.settings.json:54-61 (Stop hook, 'git add -u && git commit -m "WIP: auto-saved..."'); no settings.json-integrity check found in claude-audit-base.sh; the underlying bypass mechanism is per Claude Code's docs ('the sandbox automatically denies write access to Claude Code's settings.json files... so a sandboxed command cannot modify its own policy' — a sandbox-contingent protection), not independently live-tested in this review
- **[LOW]** Minor self-consistency gap: the kit's own dogfooded .claude/settings.json doesn't deny Edit/Write on itself, unlike the template it hands to every other project.
  - *Evidence:* .claude/settings.json:1-19 (no Edit(.claude/settings.json)/Write(.claude/settings.json) deny) vs. templates/project.settings.json:35-36

## Fit against the kit's three intentions

### As a kickoff point for new projects

Good sequencing on paper: templates/README.md's install order correctly puts 'genericize and install the managed floor' as step 1, ahead of the convenient per-repo template, and CHEATSHEET.md gives a fast, concrete 'verify it bites' ritual (read ~/.ssh/id_rsa → blocked; cat .env → blocked; force-push → prompts). But nothing gates progress on step 1 actually succeeding — a brand-new project can complete the entire kickoff flow (CLAUDE.md, evals, audit.sh, wiki) while running with zero hard floor installed, and nothing downstream will say so. For a project that starts and stays solo on one machine, this is a real, not hypothetical, gap: the single highest-leverage security decision (root-owned file placement) is also the one step that looks least like 'starting a project' and most like 'system administration,' making it the easiest step to defer indefinitely.

### As an adoption skill for existing projects

This is where the gap bites hardest and is easiest to miss. An existing team running scripts/kit-conformance.sh (ROADMAP item O) gets a clean, green adoption scorecard that is entirely compatible with zero machines on the team ever having installed the managed floor, because that check is a deliberate, permanent SKIP by design (not a temporary gap awaiting a future build). A team could roll the kit out fleet-wide, pass every conformance check, add it to onboarding docs, and be running entirely on soft, agent-editable guarantees the whole time — with the tool that's supposed to catch exactly this kind of false confidence architecturally unable to. The per-repo template layer (allow/ask/deny for git push, secret files) does port cleanly and add real value on adoption into an existing repo; the gap is specific to the hard floor, not the whole model.

### As a teaching tool beyond vibecoding

This is the strongest-written material in the whole kit for a motivated non-engineer. The five-levels table, the explicit 'who can undo it' framing repeated at every level, and the 15-minute 'try it yourself' exercise (press Shift+Tab, watch yolo refuse to engage, read .env and watch it block) build a mental model that most professional security docs don't bother building, and it is unusually calibrated about which controls are theater versus real. The miss is that the single most consequential action this same audience needs to take — hand-placing a root-owned JSON file at an unfamiliar system path — gets extensive WHY explanation but no reduction in mechanical friction (no wrapper script, no guided wizard, no post-install self-test beyond 'run claude doctor and /status yourself and notice if it's wrong'). The audience the kit explicitly names (someone stepping past vibecoding) is the audience least equipped to muscle through a sudo+JSON+path-typo-prone manual step, and also the one most likely to trust that having read the guide carefully means they're covered.

## Understand deeply vs. trust Claude

Must understand deeply: (1) that a rule's *location* — not its content — determines whether it's a preference or a wall, and specifically that Claude Code's stock, zero-config default is to allow silent, unprompted reads of any file the OS user can read, including SSH keys and cloud credentials — the entire template stack exists only to override that default, and does so only where actually installed; (2) that permission arrays merge (denies only ever accumulate and win) while some sandbox booleans do not (managed wins outright) — getting this backwards leads to believing a repo can't widen its own sandbox exclusions when in fact excludedCommands and allowRead can be appended to from a lower scope; (3) that the sandbox is Bash-only, so native Read/Write/Edit, WebFetch, and MCP tools are governed by permission rules alone, never OS-enforced; (4) that CODEOWNERS/branch protection gate the pull-request path only, not a direct push — relevant to anyone who is (or becomes) a team. What's safe to trust Claude wrote well: the specific deny-rule glob syntax and JSON key names across all three templates (independently verified against current docs — all real, current, and correctly used, including less-obvious ones like sandbox.filesystem.denyRead and sandbox.credentials.files mode:"deny"); the guarded() bash mechanism (empirically re-verified in this review); the entropy-based secret-scanning math in claude-audit-base.sh (standard truffleHog-style thresholds, correctly implemented in awk).

## Durability over the next 1–2 years

What becomes unimportant: the version-pinned specifics will age fastest and are already showing internal awareness of this (ROADMAP item M flags exactly this). The precise list of "recognized Bash readers" (cat/head/tail/sed/grep), exact JSON key names, and the 44-scenario/5-blocked red-team statistic are all snapshots of a permission engine that is visibly under active development — the docs I fetched live cite half a dozen sub-version feature changes within the 2.1.x line alone (tlsTerminate added at 2.1.199, /cd at 2.1.169, workspace-trust semantics revised three times across 2.1.196-200). If Anthropic ships a hard floor on by default (plausible, given that trajectory), the "managed floor is manual" finding could be mooted by the vendor rather than fixed by the kit. What stays vital regardless: the "a control is only as strong as the agent's inability to reach it" framing and the levels ladder itself. That question — where does this rule live, and who besides me can change it — is a property of any system where an LLM agent can edit files including its own policy file, and remains the right question even if a future harness auto-installs today's hard floor, changes the JSON schema entirely, or replaces the classifier. A future harness maturing past today's manual step would not eliminate the need to teach *why* a floor is needed — only automate satisfying it faster than a human would notice its absence.

## Missing beyond the roadmap (A–Y)

- No installer or periodic verifier for the managed floor's actual live state — a small script that reads the (typically world-readable, root-owned) managed-settings.json path and confirms its critical keys are present would be buildable without violating item O's 'SKIPPED ≠ PASS' honesty principle, since checking a file's content is different from treating its absence as a silent pass.
- No MCP-server allowlist/deny-by-default template or starter example, despite MCP being repeatedly named in prose as an equivalent-risk, unaudited surface — and despite this review's own tool list illustrating how many high-privilege MCP connectors (email, chat, drive, CRM) a real deployment can accumulate.
- No CI workflow template to run audit.sh server-side, despite the audit script's own comments arguing this is strictly superior to the client-side pre-commit hook it does ship wiring for.
- No secrets-rotation reminder or cadence mechanism — the guide recommends '~quarterly' rotation in prose but nothing schedules, tracks, or nags it, which is a strange omission given the kit's own ecosystem includes scheduling tools.
- No guidance for multi-user or shared machines (a shared dev box, a CI runner with multiple service accounts) where managed settings and per-user files interact differently, and where a second, less-trusted account materially changes the threat model the whole levels framework assumes (one user, one agent, one machine).

## Exceptional

- The 'deny rules are a sieve, only the sandbox is a general defense' analysis — verified in this review to be not just correct but current-to-the-week against live vendor docs, including subtle specifics (the literal recognized-command list, hostname-only proxy semantics, the domain-fronting caveat) that most teams either get wrong or never learn.
- The levels-tied-to-reachability model ('a control is only as strong as the agent's inability to reach it'), applied consistently rather than as a one-time slogan — every template comment, every table, and every worked example reasons from this one sentence.
- The guarded() safeguard-rot detector — a genuinely uncommon answer, for a bash audit script, to 'how would I know if my own grep-based checks silently stopped protecting anything,' and one I independently reproduced and confirmed behaves exactly as documented.

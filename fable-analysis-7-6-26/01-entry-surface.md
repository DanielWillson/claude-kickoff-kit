# Area report — Entry Surface

> **Scope:** README.md · CHEATSHEET.md · SKILL.md · glossary.md · readme-template.md · prd-template.md · styleguide.html
> **Provenance:** One of seven independent Sonnet 5 (max effort) analysis agents from the Fable-orchestrated multi-lens review of 2026-07-06. Each agent read its files in full; findings marked as reproduced were verified in scratch repos or against live vendor docs by the agent itself. Evaluated *as if all ROADMAP items A–Y were built as designed* — reported gaps are beyond the roadmap.

## Summary

This is the kit's front door and shared vocabulary: README.md (why the kit exists, the model=agent+harness framing, how to use it), CHEATSHEET.md (terse verified permission/sandbox mechanics), SKILL.md (the routing wrapper Claude Code uses to auto-invoke the kit), glossary.md (canonical term definitions), and the fill-in artifacts a fresh project inherits (readme-template.md, prd-template.md, styleguide.html), plus .gitignore for the kit repo's own housekeeping. README carries the conceptual load (agent = model + harness, directives vs. verifiers, the safety pecking order); CHEATSHEET and glossary carry the precision load (exact mechanics, exact vocabulary); the three templates are what a newcomer or an existing-project owner actually fills in and ships. This is what a human reads before ever opening claude-project-kickoff.md, and what Claude Code's skill-routing reads to decide whether to trigger the kit at all.

## Strengths

- **The core mental model (agent = model + harness; runtime vs. durable harness) is taught with unusual clarity, and the kit is candid about its own past misattribution of the idea.**
  - *Evidence:* README.md:12-32 draws the split cleanly; README.md:14-16 and 341-348 explain that 'Agent = Model + Harness' is Böckeler's framing (not Fowler's, despite his collection publishing it) and explicitly note 'this kit's own earlier drafts' got the credit wrong.
- **The safety model is taught at three deliberately different depths, explicitly cross-linked so a reader lands at the right altitude for the moment.**
  - *Evidence:* README.md:117-143 gives the plain-English pecking order; CHEATSHEET.md:8-9 says 'New here? Read the narrative field guide securing-claude-sessions.md first... this sheet is the terse reference behind it'; README.md:141-143 points to both. Most projects write one safety doc at one altitude — this kit deliberately writes three and labels which is which.
- **glossary.md's cross-references check out against the actual repository, not just against intent.**
  - *Evidence:* Spot-verified ~8 citations, all correct: glossary.md:120's '## Review' block → claude-project-kickoff.md:817; glossary.md:118's 'BEHAVIORAL EVALS' → claude-audit-base.sh:657; glossary.md:122-123's `action-risk` tag + templates/project.settings.json → claude-project-kickoff.md:796-814 and templates/project.settings.json:22-40; glossary.md:124's 'SAFEGUARD SELF-CHECK' → claude-audit-base.sh:601.
- **readme-template.md's self-verification anchor is a real, working mechanism, not an aspirational comment — it's wired end-to-end into the audit script and the kickoff checklist.**
  - *Evidence:* readme-template.md:14 `<!-- reconcile-code: PUT-PATHS-HERE -->` is the exact string claude-audit-base.sh:640-641 greps for to WARN when listed source files have moved ahead of the README; claude-project-kickoff.md:1625's checklist enforces filling it in.
- **SKILL.md's routing description uses concrete quoted trigger phrases for both flows plus an explicit negative-space exclusion — good skill-authoring craft most homegrown skills skip.**
  - *Evidence:* SKILL.md:4-11 gives quoted triggers ('run the kickoff', 'adopt the kit', 'retrofit the harness'...) for KICKOFF vs. ADOPTION and closes with 'NOT for ordinary day-to-day work in a project that already has the harness' — an explicit anti-trigger, not just a positive description.
- **The fill-in templates teach the reasoning behind each section, not just its shape, actively warning against common anti-patterns.**
  - *Evidence:* readme-template.md:44-48 warns that a hand-maintained feature checklist 'rots fastest of all and ends up lying about what's built' and redirects to the test suite/issue tracker; prd-template.md:9-11 explains why the invariants section matters ('the kickoff's §1.6 audit and CLAUDE.md extract this doc's load-bearing invariants... into greps + contract lines') instead of leaving a blank heading.

## Weaknesses

- **[MEDIUM]** styleguide.html is not a generic illustrative example — it's the maintainer's own specific personal-project design system, and README's description of it undersells what a newcomer actually receives.
  - *Evidence:* styleguide.html is 1279 lines (vs. 419 for README, the next-largest file here) of one real app's component library — favorite-avatar chips for named real people (lines 853-869: 'Cordaro', 'Daniel', 'Emma'), admin index-rebuild panels, track-queue rows, `render_album_card()`/`render_pal_chips()` references. README.md:197-199 describes it as something that 'defines the project's colors, spacing, and fonts in one place' — true of roughly the first 30 lines' token comment, not the other ~1250. Unlike its sibling templates it has no self-contained 'this is a real example — study it, then replace it' banner (contrast readme-template.md:3-8's explicit 'delete this note as you fill it in'). git log shows it entered whole in the initial commit and was never revisited, consistent with a copy-paste from an existing project rather than an authored-for-the-kit artifact.
- **[MEDIUM]** glossary.md never declares its intended audience, so whether its poor discoverability from the front door is a bug or by design can't be judged.
  - *Evidence:* glossary.md:3-4 calls itself 'the naming standard, and the docs speak it' but says nothing about who should read it. It is linked from nowhere in README.md, SKILL.md, CHEATSHEET.md, or claude-project-kickoff.md (`grep -rln glossary` across the repo hits only securing-claude-sessions.md and wiki/harness-log.md — both secondary docs — plus two files under the untracked prompts/ directory). - **[MEDIUM]** README's 'Start skeptical' list is inconsistently sourced within itself: 2 of 5 bulleted claims carry inline attribution, 3 don't, even though the lead-in claims all five are 'measured, and published.'
  - *Evidence:* README.md:41-45 names 'Anthropic' twice inline; README.md:46-49 (context/instruction-budget claim), 49-51 ('~40%' conflicting-docs claim), and 51-52 ('~17%' classifier-miss claim) name no source at all.   - *Evidence:* README.md:26-27, 93-99, and 247-249 all describe 'the instruction file' generically. ## Fit against the kit's three intentions

### As a kickoff point for new projects

Two paths exist and diverge in quality. The installed-skill path (SKILL.md) is strong: concrete trigger phrases, an explicit negative-space exclusion, and a direct jump to claude-project-kickoff.md that never touches README's density. The 'point a session at the README' path is weaker: ~230 lines of philosophy precede the first actionable instruction, and a UI project's styleguide handoff is a real personal app's design system rather than something genuinely generic — a newcomer building anything other than a personal media-sharing tool inherits a large, idiosyncratic file to mostly discard. The fill-in templates themselves are well-built once reached — short, purposeful, and (for the README template) functionally self-verifying via the reconcile-code anchor.

### As an adoption skill for existing projects

The entry-surface files correctly encode adoption as a distinct flow: SKILL.md's frontmatter names both KICKOFF and ADOPTION with separate trigger phrases, and README.md:154-160 clearly explains claude-project-adoption.md's evaluate→interview→propose sequencing and tells the reader to 'hand a session this file instead of the kickoff guide when retrofitting.' Nothing in this area actively misserves adoption; its job here is routing, and routing is correct. The overwhelm and discoverability issues (README density, glossary's unclear audience) apply equally to an adoption-minded reader — adoption doesn't get a worse deal than kickoff, just the same one.

### As a teaching tool beyond vibecoding

This is the area's best-served intention. The model+harness split, the pecking order, and the shelf-life doctrine are written for exactly the target reader — someone leveling up past vibecoding — without being condescending, and the antirez two-axis case study (README.md:68-86) models genuinely sophisticated, non-binary thinking about AI capability. Two gaps cut against the teaching specifically, though: the inconsistent inline citation practice undermines the 'start skeptical, check your sources' lesson right when it's being modeled; and never clearly naming CLAUDE.md as a literal file leaves a true beginner without the one concrete artifact-name needed to ground the abstraction.

## Understand deeply vs. trust Claude

A builder must deeply understand, from this area: (1) agent = model + harness, and specifically that the durable harness — not the model — is the thing they're building, since everything else in the kit is a special case of this; (2) the safety pecking order and why plain CLAUDE.md prose is never a boundary, which prevents a false sense of security; (3) the shelf-life doctrine (permanent/depreciating/appreciating) so they know which advice needs re-auditing at a model upgrade versus which doesn't. What they can safely trust Claude to have gotten right: the mechanical precedence/merge facts in CHEATSHEET.md (deny > ask > allow, array-merge-across-scopes, the sandbox's Bash-only scope) — lookup facts the kit itself frames as 'verified reference,' and my spot-checks of glossary.md's cross-references (all correct) suggest the kit's self-referential bookkeeping is generally sound. What should not be trusted uncritically: styleguide.html's specific content (one person's app, not a design-system endorsement) and any unattributed numeric claim in README's skeptical list until traced to LESSONS.md.

## Durability over the next 1–2 years

CHEATSHEET.md's version-pinned mechanics (explicitly '2.1.x') are the least durable content in this area and will need scheduled re-verification — the kit already names this exact problem for this exact file (ROADMAP item M/J), so it isn't a fresh gap. By contrast, the conceptual teaching — model+harness, directives/verifiers, the safety-ladder's ordering logic, the shelf-life doctrine itself — is durable because it describes the shape of the problem (a frozen model wrapped in configurable, inspectable context and tools) rather than today's flag names; a materially smarter model still needs a harness, just a differently-tuned one. glossary.md's term definitions are durable the same way, provided the maintainer keeps enforcing them — but the fact that its own build-status table already disagrees with ROADMAP's top table is a live demonstration that even a well-designed 'naming standard' erodes without active upkeep, which argues for the kit's reconcile-against-truth doctrine rather than against it.

## Missing beyond the roadmap (A–Y)

- No up-front quickstart or table of contents in README.md — a documentation-craft gap, not a feature any roadmap item names.
- No stated minimum/supported Claude Code CLI version anywhere in the entry-surface docs — CHEATSHEET.md says what it was verified against ('2.1.x') but nothing tells a newcomer with an older install that some settings/sandbox features may not exist yet; distinct from ROADMAP items J/M, which cover re-verifying after an upgrade, not stating a floor before first use.
- styleguide.html, as the kit's one worked design-system example, models no accessibility practice at all (no contrast/focus/screen-reader guidance) — a narrow but real gap given the rigor the kit spends elsewhere.
- No declared audience for glossary.md — no roadmap item addresses documentation-audience labeling as its own small discipline, the way it's correctly applied to wiki/.

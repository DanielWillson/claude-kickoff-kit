# Area report — The Kickoff Guide

> **Scope:** claude-project-kickoff.md (read in full)
> **Provenance:** One of seven independent Sonnet 5 (max effort) analysis agents from the Fable-orchestrated multi-lens review of 2026-07-06. Each agent read its files in full; findings marked as reproduced were verified in scratch repos or against live vendor docs by the agent itself. Evaluated *as if all ROADMAP items A–Y were built as designed* — reported gaps are beyond the roadmap.

## Summary

claude-project-kickoff.md is the kit's master document: ~1,678 lines (~127K chars, roughly 32K tokens) covering one-time machine hardening (Part 0), a per-project setup ritual with a nine-question intake (Part 1), ten building principles (Part 2), and an autonomous/multi-agent playbook (Part 3), plus a Quick Checklist (45 items) and a content/editorial appendix. Per the kit's own SKILL.md ("Read `claude-project-kickoff.md` — the driver"), it is read as one whole file by a fresh Claude Code session before any project code exists. Sequencing is sound for that job: tier selection and intake precede any concrete step, the security floor precedes code, and principles are internalized before the spec is requested. Where I could test claims against reality — the pre-commit-hook activation sequence, the Stop hook's auto-commit behavior, the action-risk tag/audit-script join, roughly a dozen cross-file section references — nearly all held up exactly as described, including one genuinely non-obvious git detail (index vs. disk exec bit) stated correctly. The document teaches "why," not just "do this," throughout the ritual; but it is explicitly addressed to Claude first ("Claude, read this"), and the one real structural miss is that it never treats its own bulk as a cost — it is loaded whole, in the one moment before any other context competes for space, with no acknowledgment or frugal/staged path, even though the kit applies exactly that discipline (a lean CLAUDE.md budget, the fan-out playbook built for adoption-verification) everywhere else it looks.

## Strengths

  - *Evidence:* claude-project-kickoff.md:587-591 ('git add --chmod=+x hooks/pre-commit # sets the exec bit in git's INDEX… / git checkout -- hooks/pre-commit # …and THIS sets it on disk'). Reproduced in a scratch repo: before `git checkout`, `git ls-files -s` showed mode 100755 in the index while `ls -l` on disk still showed `-rw-r--r--`; after `git checkout -- hooks/pre-commit`, disk mode became `-rwxr-xr-x`, and a real `git commit` with a staged fake `.env` was then actually BLOCKED by the wired hook (exit 1) — the full documented chain works end-to-end.
- **The action-risk taxonomy is a verifiable design, not decorative prose: a shared `action-risk` tag ties a CLAUDE.md table row to a real permissions.ask/deny rule, and a script proves the join exists.**
  - *Evidence:* claude-project-kickoff.md:633-692 (§1.3c) and the CLAUDE.md skeleton table with its `<!-- action-risk -->` marker at :796-816; cross-checked against claude-audit-base.sh:264-297, whose grep-based join logic (keys on the tag, ignores commented-out rules, WARNs rather than FAILs) matches the doc's description exactly, and whose own comments honestly flag the check as structural-only ('not on semantic action->rule correctness') — the same honest caveat the kickoff doc itself states at §1.6c.

## Weaknesses

- **[MEDIUM]** The Stop hook's fallback-commit failure message names the wrong cause for a common, hook-unrelated failure mode.
  - *Evidence:* claude-project-kickoff.md:284, the Stop hook's command string prints 'Auto-commit blocked (a hook rejected it - likely a staged secret); nothing committed, changes still staged' whenever `git commit` returns nonzero. Reproduced in a scratch repo: with only an untracked new file present (no other changes), `git status --porcelain` is non-empty so the hook's `if` fires, but `git add -u` does not stage an untracked file, so `git commit` fails with git's own message 'nothing added to commit but untracked files present' (exit 1) — no pre-commit hook needs to exist for this (the §1.3b secret hook is opt-in, off by default outside Hardened) and nothing was ever staged, so both clauses of the printed message are wrong for this cause. The exit-nonzero-instead-of-silent-success behavior itself is correct and valuable; only the diagnostic text is misleading.
- **[MEDIUM]** The guide never treats its own length as a context cost, even though the kit applies exactly that discipline everywhere else it looks.
  - *Evidence:* claude-project-kickoff.md is 1,678 lines / ~127,503 characters (~32K tokens by a chars/4 estimate) and per SKILL.md:29 ('Read `claude-project-kickoff.md` — the driver') is read whole, in one sitting, by the session that must then execute it faithfully; the doc's own 'don't read cover-to-cover' note (lines 39-45) applies only to companion files pulled in on demand, not to the guide's own bulk, which has no chunked or staged reading path. Nothing in the file acknowledges this footprint, despite the kit elsewhere treating context as a managed budget (the CLAUDE.md lean-line doctrine at :847-856; the fan-out playbook at §1.6c built specifically because, per ROADMAP item O, 'one session reading all of it will blow its context') — that fan-out discipline is never pointed at the kickoff-execution session itself, a comparable one-session/reads-a-lot situation for a different purpose (building rather than verifying). This is not a defect in the Lean/Standard/Hardened tiering itself, which is real and honestly scoped for what work gets done (§1.0's table correctly reduces Lean to §1.1+§1.3+§1.5); the gap is that the file loads whole regardless of tier, so a Lean session pays nearly the same reading cost as a Hardened one for a much smaller resulting workload.
- **[LOW]** The kit's own glossary is never linked from the file most likely to need it, or from the README that funnels readers to that file.
  - *Evidence:* grep -ni "glossary" against claude-project-kickoff.md and README.md returns zero matches in both (confirmed via exit code 1 on both greps), even though glossary.md is linked from claude-project-adoption.md and wiki/harness-log.md. Kickoff.md relies throughout on kit-coined vocabulary (harnessability at :1167-1169, 'action-risk' at 11 places, BRIEF.md-'the brief' at :737-739, 'the floor' used dozens of times) that a reader taking 'the next step beyond vibecoding' (the kit's stated intention 3) would benefit from a glossary pointer for.
- **[LOW]** Part 0's machine-wide hardening reads as an unconditional prerequisite, with no prominent on-ramp for the most casual persona the kit's own first intention names.
  - *Evidence:* claude-project-kickoff.md:51-52 states 'Do this ONCE per machine, before the per-project ritual' with no qualifier; the only acknowledgment that a project could proceed without it is one buried parenthetical at :333 ('doubles as a fallback if you skip Part 0'), 280+ lines later and phrased as a side note about allowlist strictness, not as guidance offered to a first-time or 'silly'-project user (the kit's own stated range in intention 1) deciding how to start.
- **[LOW]** The document is structurally ritual-first and principle-second for a file the README funnels newcomers to read first, and it does not itself redirect a learner to the more teaching-oriented companion.
  - *Evidence:* Part 2's transferable principles (Simple Made Easy, dependency restraint, etc.) begin at line 1140, after ~1,140 lines of Claude-Code-specific mechanics (JSON permission schemas, sandbox internals, bypassPermissions semantics) in Part 0-1; `grep -n "LESSONS"` against the file returns exactly one hit (line 1009), an incidental citation for 'the Rule of Five,' not a 'read this first to understand' pointer — that redirection exists only in README.md ('Start here if you want to understand before you install'), which a reader following README's other, more prominent instruction to kickoff.md ('Read this first; it drives everything else') may not separately notice.
- **[LOW]** One cross-reference is imprecise in an otherwise very accurate cross-referencing apparatus.
  - *Evidence:* claude-project-kickoff.md:1113 attributes 'parallel read-only verifiers + a single writer' to 'Part 3.1' (item 1, 'Fan out only for genuine fan-out'), but item 1's text never mentions single-writer discipline — that idea belongs to item 4 ('Give every file exactly one writer: partition by directory...'). Minor, and the only slip found among roughly a dozen cross-references checked.
- **[LOW]** Roadmap item M's proposed fix ('date-stamp version-specific facts + the re-verification habit') is under-specified as a design: it names no machine-detectable staleness convention, so it doesn't extend to this file the same discipline the kit gives audit greps (item H's anchor check) or wiki pages (the `verified:` clock).
  - *Evidence:* grep for '(verified' / 'verified mid-2026' in claude-project-kickoff.md returns exactly 2 hits (lines 850 and 1423) against dozens of undated Claude-Code-mechanic claims in the same file (e.g. :431-437 sandbox local-binding behavior, :702-711 `claude doctor`/`/status` key-stripping behavior, :403-408 bypass-mode semantics, :1419-1428 worktree/subagent feature availability) — there is no visible marker distinguishing 'this describes today's Claude Code, re-check on upgrade' from 'this is a permanent principle,' so M's design, as scoped, leaves the file's own prose exposed to exactly the 'agents follow stale docs with total confidence' failure mode the kit's own README opens with.

## Fit against the kit's three intentions

### As a kickoff point for new projects

Strong on its primary job. Tiering (Lean/Standard/Hardened, §1.0) is real and honestly scoped for what work gets done — Lean genuinely skips the wiki and audit, and "if non-throwaway"/"scale by tier" gates recur consistently through §1.6a/b/c and the 45-item Quick Checklist (214 total list items in the document), not just in the tier table. Intake-first design (§1.0a, nine questions answered once) avoids interrupting the ritual repeatedly, and the safety floor (§1.3) is copy-paste-then-adapt with concrete bite-tests ("read ~/.ssh/id_rsa -> blocked"); I reproduced several of the shipped snippets (the pre-commit hook, the hooksPath activation sequence) end-to-end in a scratch repo and they work exactly as documented. Where it misses: the casual end of intention 1's own stated range (a "silly" or first-time project) still meets Part 0's full machine-hardening ceremony stated as an unconditional prerequisite, with only one buried parenthetical (line 333) revealing it is actually skippable in favor of the per-repo floor alone; and the Stop-hook auto-commit snippet every tier ships (line 284) prints a misleading message on a failure mode I reproduced that has nothing to do with any hook (an untracked new file left uncommitted at session end) — no work is lost, but the signal misdirects debugging effort at exactly the unattended moments this intention leans on.

### As an adoption skill for existing projects

Kickoff.md correctly stays out of claude-project-adoption.md's lane: its own header states it is "not a CLAUDE.md" and points retrofits to the adoption doc, and SKILL.md repeats the same fork. The infrastructure adoption.md reuses from this file — the settings block (§1.3), the CLAUDE.md skeleton (§1.5), the audit (§1.6), and the scorecard/evals/conformance triad (§1.6a-c) — is exactly the material I verified to be accurate and well cross-referenced from the kickoff side, so the shared foundation adoption.md builds on is solid. I did not evaluate claude-project-adoption.md's own re-sequencing, which is outside this area.

### As a teaching tool beyond vibecoding

Moderate, and secondary to LESSONS.md by the kit's own design — but that hand-off depends on the reader finding it, and this file does not do the pointing itself. Kickoff.md mentions LESSONS.md exactly once (line 1009), as an incidental citation rather than a "read this first" signpost; the actual redirection ("Start here if you want to understand before you install") lives only in README.md, while README also tells the reader to read kickoff.md first ("it drives everything else") — a reader who follows the second instruction without separately catching the first gets Claude-Code-specific JSON permission schemas and sandbox internals (Part 0-1, ~1,140 lines) before Simple Made Easy or dependency restraint (Part 2), the reverse of how a person building transferable mental models would want the material ordered. The kit's own glossary is never linked from here despite heavy use of kit-coined terms. What the file does teach, it teaches well: reasoning is attached to nearly every consequential ritual choice rather than stated as fiat (the several "Why these specific denies" / "Why the deny list — not denyWrite — is the real floor" asides), and the layered-security model is genuinely internalizable because it is restated economically at each point of use rather than buried once or repeated wholesale.

## Understand deeply vs. trust Claude

Deep, irreducible understanding needed: the four-tier control hierarchy (OS sandbox as the only true wall; deny/ask as deterministic backstops; the classifier as probabilistic; CLAUDE.md prose as advice) — without internalizing this, "protected" and "reads as protected" can't be told apart, which is the exact trap §1.3a's residual-risk section (lines 460-531) is built to prevent. The Bash-only scope of the sandbox — that native Edit/Write/WebFetch/MCP never cross it, which is precisely why the floor needs parallel permission-layer Read/Write/Edit denies alongside sandbox denyWrite (kickoff:307-327) — is a common misconception the doc corrects explicitly and a human must hold onto when extending the floor to a new project. The reversibility x reach action-risk axes (§1.3c) require judgment the kit cannot supply: it gives a taxonomy and worked examples, but classifying each new action a specific project introduces is the human's job, not a lookup. And commit granularity as the review surface (Principle 4) — in an auto-mode world, if a human doesn't personally enforce small logical commits, the mechanism that was supposed to make after-the-fact review possible quietly disappears, and nothing in the harness catches that omission. Safe to trust Claude got right: the boilerplate JSON syntax and shipped scripts — I ran several directly (the pre-commit hook, the Stop hook, the hooksPath activation sequence) and they work exactly as described, including one non-obvious git detail (index vs. disk exec bit) most engineers would not get right unassisted; the exact wording of the CLAUDE.md skeleton; and the specific numeric citations, which the document itself models treating as directional rather than gospel ("the viral 'keep it under 50 lines' rule has no primary source — ignore it," line 852).

## Durability over the next 1–2 years

The doc's own three-shelf-life framework (permanent / depreciating / appreciating, lines 311-330) is a genuinely sound answer to this question, and I'd mostly endorse its placement: the security floor, reconcile-against-code, and independent verification are argued to be permanent because their force comes from properties of the world (a self-report is a claim regardless of model quality; a fact outside context doesn't exist regardless of intelligence), not from today's model's specific weaknesses — that reasoning holds up mechanistically and should still hold in 1-2 years. What should genuinely shrink: prescriptive step-by-step coaching and dated mechanic claims keyed to Claude Code 2.1.x specifics (exact `claude doctor`/`/status` behavior, "the sandbox blocks binding local sockets by default," specific auto-mode classifier drop-lists) — these are properties of today's runtime harness, not eternal, and the file dates only 2 of dozens of such claims (see weaknesses), so there is no reliable way to tell which sentences are due for re-audit after an upgrade. What should NOT shrink, and which the file already argues correctly: the builder/judge split (self-grading is structurally unreliable, not just today's-model unreliable) and the general context-is-a-budget doctrine — though this file's own failure to apply that doctrine to itself (loaded whole, uncosted) is a case where the kit's "do as I say" and "do as I do" have already diverged, and I would expect that specific tension to matter more, not less, over the next two years as the kit itself keeps growing by accretion (visible in the proliferation of lettered subsections, 1.3a/b/c, 1.5a/b/c, 1.6a/b/c) — bigger context windows don't repeal recall-degrading-with-context, which the kit's own README treats as a durable finding rather than a today-only artifact.

## Missing beyond the roadmap (A–Y)

- A context-budget mitigation for the kickoff-EXECUTION session itself, distinct from item O's adoption-verification fan-out: no roadmap item proposes chunking, staging, or otherwise reducing the cost of a fresh session reading this 1,678-line driver file whole before any setup work begins.
- glossary.md exists, is well-built, and is linked from claude-project-adoption.md and the wiki, but is never linked from claude-project-kickoff.md or README.md (verified via grep) — a near-zero-cost fix with no roadmap item covering it.
- No fully-worked, compact 'here is a Lean kickoff end-to-end' example distinct from the implicit Standard-oriented default narrative the rest of Part 1 tells — a Lean-tier reader has to synthesize their actual path themselves by subtracting sections rather than following one worked instance.
- No guidance for re-running any part of the ritual when intake answers change mid-project (solo project acquires a second committer; a throwaway gets kept) beyond the abstract 'the maturity trigger adds hardening' line — distinct from item J, which is scoped specifically to Claude Code version upgrades, not to intake-answer drift over a project's life.

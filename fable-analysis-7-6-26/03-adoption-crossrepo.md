# Area report — Adoption & Cross-Repo Layer

> **Scope:** claude-project-adoption.md · HARNESS_LOG.md · scripts/kit-conformance.sh · prompts/
> **Provenance:** One of seven independent Sonnet 5 (max effort) analysis agents from the Fable-orchestrated multi-lens review of 2026-07-06. Each agent read its files in full; findings marked as reproduced were verified in scratch repos or against live vendor docs by the agent itself. Evaluated *as if all ROADMAP items A–Y were built as designed* — reported gaps are beyond the roadmap.

## Summary

This area covers intention #2 (existing projects adopting the kit): claude-project-adoption.md (the retrofit sequencing guide), HARNESS_LOG.md (the portable harness-change-log template plus its cross-repo-learning teaching, taught in kickoff §1.6a), scripts/kit-conformance.sh (the item-O deterministic adoption verifier), and four prompts/build-*.md files that were used to build items O/R/V/X into the kit. Together these implement "verify adoption with a verifier, not a self-report" and a bounded cross-repo learning capability. I read all seven files in full, ran `bash -n` on the script, and empirically stress-tested it against ~10 fixtures I built under $TMPDIR (floor-only, fully-conformant, each FAIL-class broken, each WARN-class degraded, an AGENTS.md-only repo, a truncated settings.json, and the kit repo itself unmodified). Note: the repo was modified concurrently during this review — a fifth prompts/build-Y file appeared mid-session, confirming the directory is actively growing with no index.

## Strengths

- **The three-class exit model (FAIL only what no correct adoption could omit / WARN what a lean adoption may skip / SKIP what can't be verified / nonzero only on FAIL) is empirically sound, not just well-argued in prose. I built a floor-only fixture (CLAUDE.md + deny floor + valid audit.sh, nothing else) and it reported zero FAIL, exit 0; a fully-conformant fixture passed clean; each of the three FAIL-class artifacts (missing CLAUDE.md, missing .claude/settings.json, syntax-broken audit.sh) independently flips to FAIL/exit 1 when broken; commenting out (vs. removing) the deny rules correctly WARNs rather than FAILs; and running with no argument inside a seeded project's scripts/ correctly resolves TARGET via the $ROOT default.**
  - *Evidence:* scripts/kit-conformance.sh lines 41-58, 209-220; verified live via fixtures under $TMPDIR (floor-only -> PASS:4 WARN:4 FAIL:0 exit 0; each break -> FAIL + exit 1)
- **Predicate reuse between the audit and the conformance script is real, not just claimed — the ACTION-RISK GATES block is a line-for-line logic match.**
  - *Evidence:* scripts/kit-conformance.sh lines 111-122 vs. claude-audit-base.sh lines 285-296 — same marker-join, same non-comment filter, same skip message, confirmed by direct side-by-side read
- **claude-project-adoption.md is genuinely evidence-driven for the hard real-world case, not generic retrofit advice — it treats an already-tracked secret as a live incident with an ordered remediation (gitignore, git rm --cached, rotate the credential, add the audit FAIL), not a checklist line, and separately protects other committers from being ambushed by a silent settings change.**
  - *Evidence:* claude-project-adoption.md lines 81-88 ("A tracked secret is a live incident, not a checklist item" ... "Don't break the team's flow")
- **Citation discipline holds when checked: the script cites the adoption guide by raw line number twice, and both resolve to exactly the quoted text.**
  - *Evidence:* scripts/kit-conformance.sh lines 186/198 ("adoption §159", "adoption §133") match claude-project-adoption.md lines 158-159 and 133-134 verbatim, confirmed by direct read
- **HARNESS_LOG.md's field-level schema is genuinely lean and grep-verified to work in practice — the six bold-label fields are reliably extractable even though the header-date proof fails against the still-templated anchor (see weaknesses).**
  - *Evidence:* grep -nE '^- \*\*(change|rationale...|origin)' HARNESS_LOG.md returns all six field lines cleanly, reproducing prompts/build-X-full-cross-repo-learning.md's own verification step 2

## Weaknesses

- **[HIGH]** kit-conformance.sh never validates that .claude/settings.json is syntactically loadable JSON — it only checks file presence plus a text grep for deny patterns. A truncated/corrupted settings file (missing closing braces, would almost certainly fail to load in Claude Code, voiding every deny rule, the push gate, and the Stop hook) still gets a clean PASS. This directly contradicts claude-project-adoption.md's own Definition-of-Done, which promises the floor is 'proven to bite... a denied secret read actually blocks' and names this exact script as the machine check of that promise. The same script DOES syntax-check its sibling artifact (bash -n on scripts/audit.sh) three sections above — so the omission reads as an inconsistency, not a considered scope decision.
  - *Evidence:* scripts/kit-conformance.sh lines 137-147 (presence+grep only) vs. lines 164-173 (bash -n on audit.sh); claude-project-adoption.md lines 155-156, 162-163 ("proven to bite... a denied secret read actually blocks" / "Verify that with a verifier... bash scripts/kit-conformance.sh... is the machine check"); reproduced live: a hand-truncated settings.json with unbalanced braces (confirmed invalid via python3 json.load) still printed "per-repo deny floor present — active secret-READ deny"
- **[HIGH]** The action-risk marker mechanism that item R built and item O (via claude-audit-base.sh's verbatim predicate) certifies requires an inline trailing `//` comment on a live JSON array element inside .claude/settings.json — a construct that is objectively invalid JSON (confirmed independently via Python's json parser) and that open Claude Code feature requests (anthropics/claude-code #29370, #12688, #17968, found mid-2026) suggest is not reliably parsed as JSONC today. templates/project.settings.json ships exactly this construct as its worked example. I could not directly test Claude Code's own parser or confirm whether a load failure is silent or loud, and the kit's own live, committed .claude/settings.json sidesteps the risk (it uses only a leading comment banner before the opening brace, never an inline comment on an array element) — so the one settings.json known to be in real use never exercises the riskier form the action-risk mechanism specifically demands. I built a fixture using exactly the prescribed inline-tag form and confirmed kit-conformance.sh gives it a clean 'action-risk gates wired' PASS while Python's json parser rejects the identical file. This is a root cause that likely belongs to whichever area owns templates/project.settings.json / kickoff §1.3 and §1.5 — flagging here because it is what lets my assigned script (kit-conformance.sh) hand out a false PASS on the one check item O added specifically to prove 'prose is not a boundary.'
  - *Evidence:* prompts/build-R-action-risk-tiers.md line 52 ("a paired comment on each action-risk ask/deny rule"); templates/project.settings.json lines 20-25, 38-40; claude-project-kickoff.md lines 239-274 (fenced JSON block with inline `//` comments presented as literal file content); scripts/kit-conformance.sh lines 111-122; reproduced live with a fixture + python3 json.load
- **[MEDIUM]** kit-conformance.sh hardcodes CLAUDE.md (line 64: CLAUDE_MD="$TARGET/CLAUDE.md") with no fallback to AGENTS.md, contradicting the kit's own stated policy that 'Claude reads either' and that a project may keep AGENTS.md as the sole physical file with CLAUDE.md merely symlinked to it (or vice versa). A project that correctly adopted AGENTS.md-only, without adding the symlink the adoption guide nudges toward, gets a hard FAIL that also silently skips the routing/budget/reviewer checks (which the script gates on CLAUDE.md's presence). ROADMAP §4 names AGENTS.md as a real, actively-governed cross-tool convention, so this gap is likely to bite more often over time, not less.
  - *Evidence:* scripts/kit-conformance.sh line 64, 72-78; claude-project-kickoff.md line 727 ("Claude reads either"); claude-project-adoption.md lines 117-119 ("keep one physical file, symlink the other name"); reproduced live with an AGENTS.md-only, no-symlink fixture -> hard FAIL, exit 1, cascading skip of the routing/budget/reviewer rows
- **[MEDIUM]** HARNESS_LOG.md's shipped anchor entry keeps the placeholder <YYYY-MM-DD> in its header (correctly, per the kit's own 'never a filled-in instance' rule) — but this means the exact machine-legibility proof prompts/build-X-full-cross-repo-learning.md specifies (a header regex requiring real digits) returns zero matches against the file as shipped; the field-level grep does work, so the schema is legible in practice once a real date lands there. More importantly, nothing anywhere (kit-conformance.sh and claude-audit-base.sh both explicitly, by documented design, exclude HARNESS_LOG.md from their rosters) checks that an adopting project ever actually replaces the placeholder with a real kit version/commit. Item Y's whole mechanism depends on that stamp being real; left as a placeholder — a very plausible outcome for an unenforced TODO — Y silently has nothing to diff against and nothing in the kit would ever flag it.
  - *Evidence:* HARNESS_LOG.md line 70 ("## <YYYY-MM-DD> — Adopted..."); prompts/build-X-full-cross-repo-learning.md verification step 2 (the header regex); reproduced live: grep -nE '^## [0-9]{4}-[0-9]{2}-[0-9]{2} — ' HARNESS_LOG.md returns nothing; claude-audit-base.sh lines 336, 351 ("HARNESS_LOG.md are in NEITHER clause BY DESIGN")
- **[MEDIUM]** The prompts/ directory (the four build specs) is a good idea executed as scaffolding debt today: it is untracked in git (git status shows '?? prompts/'), referenced from nowhere else in the kit (README, ROADMAP, glossary, wiki all have zero hits for 'prompts/'), has no stated convention for what belongs there or how long files live, and no marker distinguishing a completed build from a pending one. Concretely, two of the four files (build-R, build-V, both dated 2026-07-04) open with the stale absolute path /Users/danielwillson/dev/claude-harness-kit, while the two later files (build-O, build-X, both 2026-07-06) correctly say claude-kickoff-kit — the repo was renamed between the two dates and the earlier saved prompts were never updated. The old directory no longer exists, so re-running an old prompt verbatim would fail loudly rather than silently corrupt a stale checkout, but it demonstrates the pattern has no rot-check analogous to the kit's own item H, applied to itself.
  - *Evidence:* prompts/build-R-action-risk-tiers.md line 1, prompts/build-V-name-the-reviewer.md line 1 (both "claude-harness-kit") vs. prompts/build-O-adoption-check.md line 1, prompts/build-X-full-cross-repo-learning.md line 1 (both "claude-kickoff-kit"); repo-wide grep for "prompts/" outside the directory itself returns zero hits; git status at session start shows prompts/ untracked
- **[MEDIUM]** Cross-repo learning's 'human supplies the source' rail treats a local path and a live URL as equally trust-conferring ('a path or URL they give you'), but a URL is a weaker link: the human may not have actually read the content behind it, and the content can change between when they endorsed the link and when the agent fetches it (no pinning/commit-SHA guidance). The complementary 'propose, never auto-apply' rail is otherwise sound and reasonably backstopped — if a hostile borrowed log tried to get the agent to act rather than merely propose, the kit's pre-existing deny/sandbox floor (not anything specific to this feature) would still be the thing that stops it, since there is no settings.json-level gate for 'reading a file and drafting a proposal.'
  - *Evidence:* claude-project-kickoff.md line 1019 ("a path or URL they give you"); prompts/build-X-full-cross-repo-learning.md line 27 ("a local path, or a URL they give you")
- **[LOW]** kit-conformance.sh declares TMP="${TMPDIR:-/tmp}" (apparently copied from harness-metrics.sh's boilerplate) but never references $TMP anywhere in the script.
  - *Evidence:* scripts/kit-conformance.sh line 44; grep '\$TMP\b' returns no hits in the file
- **[LOW]** The script's own explanatory comments cite the adoption guide by raw line number rather than a stable anchor; both citations resolve correctly today, but nothing re-verifies them if claude-project-adoption.md is edited above those lines — the same silent-drift failure mode item H's safeguard-rot check exists to catch elsewhere, here applied to a human-facing comment rather than a functional check, so the practical blast radius is a misleading comment, not a broken gate.
  - *Evidence:* scripts/kit-conformance.sh lines 186, 198 ("adoption §159", "adoption §133")

## Fit against the kit's three intentions

### As a kickoff point for new projects

Secondary fit (this area is mainly about adoption), but real: kit-conformance.sh is explicitly meant to run at the tail of a fresh greenfield kickoff too (§1.6c: "run it after kickoff/adoption and periodically thereafter"), so its bugs land on day one of intention #1 as well. A brand-new project that chose AGENTS.md over CLAUDE.md, or whose settings.json got hand-edited into something slightly malformed before the first check, would see a spurious FAIL or a false-confidence PASS at the exact moment a new user most needs the tool to be credible. The exit-model design itself, though, is a genuinely good fit for kickoff's tier-awareness (Lean/Standard/Hardened) since it never punishes a throwaway for skipping the optional roster.

### As an adoption skill for existing projects

This is the area's primary intention, and the prose half (claude-project-adoption.md) serves it well: it correctly inverts the greenfield posture ("first job is to not break what works"), fans out the evaluation sweep for large repos, and gives concrete, ordered handling for the hardest real case (a secret already tracked in git history) rather than generic advice. Where it misses is the payoff that makes the whole thing trustworthy at scale — "verify with a verifier, not by re-reading a list" is only as good as the verifier, and I found the verifier gives a false PASS on the single highest-stakes row (the security floor) when that file is malformed, and a false FAIL on a legitimately-adopted AGENTS.md-only repo. For a real messy existing repo, expect the guided walkthrough to work well and the automated "did it actually stick" check to need a human's judgment on exactly the two things it was built to remove judgment from.

### As a teaching tool beyond vibecoding

Strong as a worked example, weaker as an explicit lesson. kit-conformance.sh's design — three exit classes, "structural not semantic, named not faked," reusing a sibling script's predicate rather than re-deriving it — is a better model of honest verifier-building than most real CI tooling a learner will encounter, and the build prompts are an unusually explicit, readable record of spec-driven feature development (a learner could read prompts/build-O-adoption-check.md end to end and see exactly how "plan, get the hard part right, verify adversarially, report honestly" plays out on a real task). What it does not teach explicitly: the mechanical fact I had to discover empirically — that a security floor is a parseable artifact whose failure is often silent, and that "the check passed" does not mean "the thing the check describes is actually true." That is precisely the kind of concrete, humbling lesson a person stepping beyond vibecoding needs, and the kit currently only demonstrates it by accident (in my testing) rather than teaching it on purpose.

## Understand deeply vs. trust Claude

Must understand deeply: (1) that a permissions/settings file is a real artifact with a real parser, and its failure mode is typically silent, not a warning banner — "the file exists and contains the right-looking text" is not the same claim as "the protection is active," and a builder who conflates them will trust a floor that isn't there; (2) the FAIL/WARN/SKIP distinction and specifically that SKIPPED ≠ PASS — treating a clean scorecard as a green light without reading which rows were skipped is a real failure mode this kit itself names but cannot fully prevent; (3) that "propose, never auto-apply" in the cross-repo-learning design is a request made to the model, not a settings-level wall — a builder who wants a hard backstop for a genuinely untrusted source needs to know that and keep the review step manual. What can safely be trusted as Claude having written well: the exit-code arithmetic and PASS/WARN/FAIL bucketing logic in kit-conformance.sh itself (I adversarially stress-tested it across a dozen fixtures and it held completely); the day-to-day sequencing prose in claude-project-adoption.md, which is internally consistent and cites real, resolvable line numbers.

## Durability over the next 1–2 years

The stated reason for Part 2's fan-out ("one session reading all of it will blow its context") is explicitly a today's-context-window argument; if context windows grow substantially over 1-2 years, that specific justification weakens and a single session may again comfortably hold the whole kit plus a project. But the deeper reason to keep an independent, fan-out-style read stays vital regardless of context size: a model with unlimited context can still rubber-stamp its own self-report (the "Confident and Wrong" finding the kit's own ROADMAP already cites is not a context-limitation problem, it's a verification-incentive problem), so "don't trust a self-report, get an independent read" survives even as "can't fit it in context" stops being the reason why. The concrete bugs I found (no JSON-validity check, hardcoded CLAUDE.md filename) are pure script-logic gaps that a more capable model running the same unmodified script will reproduce identically — they need a rewritten check, not a smarter model, to fix. Deterministic action-gating (item R's spine: "a control is only as strong as the agent's inability to reach it") is a property of security architecture, not model capability, so it stays vital indefinitely. Cross-repo learning's value is likely to appreciate specifically as the fleet of kit-derived repos grows (a network effect independent of model quality) — the kit's own wiki entry argues this and I think the mechanism is sound. The prompts/ pattern's core idea (durable, falsifiable build specs you can audit after the fact) stays useful regardless of model changes; its current staleness problem is a process/indexing gap that better models will not automatically fix without someone building the equivalent of item H pointed at that directory.

## Missing beyond the roadmap (A–Y)

- No JSON/config-validity check anywhere in the kit's verification stack for .claude/settings.json — a cheap, high-value gap given the audit script already sets the precedent of syntax-checking its sibling artifact (bash -n on audit.sh); could be closed the same way (prefer jq if present, else python3 -c json.load, else a loud SKIP naming that validity could not be confirmed) without adding a hard dependency.
- No enforcement or even a reminder mechanism that HARNESS_LOG.md's placeholder anchor (<kit-version>, <commit-sha>, <YYYY-MM-DD>) actually got replaced with real values before item Y is built on top of it — a single unfilled TODO silently breaks the entire cross-repo-learning/Y chain with no check ever firing.
- The fan-out coordinator's own merged scorecard is itself an unverified self-report — item O's central 'don't trust a self-report, verify with a verifier' principle has no teeth once turned on the fan-out's own orchestration layer; nothing requires or checks evidence that N genuinely separate, slice-loaded sub-agent contexts were actually used rather than one context serially roleplaying N reviews (this review task's own multi-agent structure is itself a live instance of the pattern, with the same absence of a meta-check).
- No convention, index, or rot-check for the prompts/ directory itself, despite item H existing for exactly this class of silent drift elsewhere in the kit; two of four files are already stale on a basic fact (repo path) within 48 hours of each other with zero detection.

## Exceptional

- The exit-model design (FAIL only what no correct adoption could omit / WARN what a lean-but-correct project may legitimately skip / nonzero only on FAIL) is a genuinely sharp, transferable answer to how a conformance checker stays honest across wildly different project scales without becoming either useless (never fails) or a nuisance that gets disabled (fails everything) — and it is not just argued in prose, it held up completely across every fixture I adversarially threw at it.
- The discipline of reusing the exact same predicate, verbatim, between claude-audit-base.sh and scripts/kit-conformance.sh rather than re-deriving a parallel implementation is real engineering hygiene that most codebases talk about and few practice; I confirmed the two blocks are a line-for-line logic match.
- claude-project-adoption.md's ordered handling of an already-tracked secret (gitignore -> git rm --cached -> rotate the credential -> add the audit FAIL, done before any other adoption work) is more concrete and operationally correct than most retrofit guidance, which typically stops at .gitignore and never mentions that history still holds the secret.

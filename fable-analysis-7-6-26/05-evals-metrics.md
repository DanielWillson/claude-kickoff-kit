# Area report — Evals & Metrics Layer

> **Scope:** evals-template/ · claude-eval-base.sh · scripts/harness-metrics.sh
> **Provenance:** One of seven independent Sonnet 5 (max effort) analysis agents from the Fable-orchestrated multi-lens review of 2026-07-06. Each agent read its files in full; findings marked as reproduced were verified in scratch repos or against live vendor docs by the agent itself. Evaluated *as if all ROADMAP items A–Y were built as designed* — reported gaps are beyond the roadmap.

## Summary

This area is the kit's judgment-and-ROI measurement layer: a behavioral-eval suite (`evals-template/` + `claude-eval-base.sh`, roadmap items A/I) that checks agent *judgment* the way tests check code, via two grade types — golden-output (string equality, no live model needed to grade) and rubric/LLM-judge (a fresh agent grades against a blunt checklist) — plus a harness scorecard (`scripts/harness-metrics.sh`, item B) that snapshots a couple of repo-derivable numbers and appends them to a trend log, pairing with the qualitative `HARNESS_LOG.md`. Together they're meant to answer "is the agent's judgment holding up, and is my safety net actually paying off" rather than assuming both. The fixture format is a stack-agnostic markdown+frontmatter schema (input/expected/rubric/required_sources/forbidden_actions/approval_class) parsed with grep/sed only, deliberately avoiding jq/python3 dependencies. I read every file fully and ran both scripts under real and stubbed conditions (golden PASS/FAIL, rubric PASS/FAIL, malformed fixtures, empty dirs, and a simulated adopting project) rather than only reading the source.

## Strengths

- **The golden-vs-rubric noise caveat is unusually honest and consistently propagated, not a bolted-on disclaimer.**
  - *Evidence:* evals-template/README.md lines 30-35: 'LLM-as-judge is noisy: documented bias, roughly 6-percentage-point swings from the evaluation infrastructure alone, and models can detect they are being evaluated... treat this suite as a smoke alarm, not a lab scale.' The identical framing (same 6pp figure, same 'smoke alarm' phrase) recurs verbatim in claude-eval-base.sh's header comment (lines 12-14) and claude-project-kickoff.md §1.6b — it shapes the tool's actual design (golden preferred, rubric blunt) rather than just disclaiming after the fact.
- **Explicit, code-verified honesty about what the runner mechanically enforces vs. what is schema-only.**
  - *Evidence:* evals-template/README.md lines 71-79 states required_sources/forbidden_actions/approval_class 'are schema fields the project grows into... the base runner does not mechanically enforce them.' I confirmed this is true of the actual code: fm() in claude-eval-base.sh is only ever invoked for the 'grade' key (line 87); nothing reads approval_class, required_sources, or forbidden_actions. Most tooling lets a schema field's presence imply enforcement — this doesn't, and the claim matches the code exactly.
- **harness-metrics.sh refuses to fabricate unmeasured data — a genuinely rare form of restraint in a scorecard tool.**
  - *Evidence:* Header comment lines 18-24: 'do NOT over-instrument... never a fabricated zero.' Backed by code: manual() (line 53) prints literal 'human note required' for 7 of ~9 tracked dimensions (lines 106-112) rather than a silent 0 that would misleadingly read as 'zero incidents.'
- **Careful defensive bash in the scorecard's trend arithmetic, correctly anticipating a real edge case.**
  - *Evidence:* delta() (lines 60-69) guards both operands with a numeric regex before arithmetic, and uses the `10#` base-10-forcing prefix. I verified empirically that without it, a leading-zero value like '08' in a hand-edited log line hard-errors bash arithmetic ('value too great for base') — the script correctly defuses a real, non-obvious failure mode.
- **The stack-agnostic parser and runner genuinely work on their happy paths, and the 'stub proves the harness with no live model' claim is true for the PASS case.**
  - *Evidence:* I ran claude-eval-base.sh against both shipped example fixtures with stubbed EVAL_CMD/EVAL_JUDGE_CMD (golden PASS and rubric PASS) and got correct PASS/PASS output with exit 0 — the documented no-live-model self-test capability holds for the success path.

## Weaknesses

- **[HIGH]** claude-eval-base.sh crashes the ENTIRE runner on any golden-eval failure, instead of reporting FAIL — reproduced on macOS's stock bash and on current Homebrew bash under a standard UTF-8 locale.
  - *Evidence:* Line 101: `efail "$name  [golden]  expected «$expected»  got «$candidate»"`. The unbraced `$expected`/`$candidate` immediately followed by the closing guillemet '»' (UTF-8 lead byte 0xC2) causes bash's identifier scanner to consume that byte into the variable name, corrupting it; under `set -u` (line 46) this raises 'unbound variable' and aborts the whole script. I reproduced this identically on macOS system bash 3.2.57 and on Homebrew bash 5.3.15 (both under LANG=en_US.UTF-8) with a minimal repro and with the real shipped file (via a mismatching stub builder). Not verified on Linux/glibc — scoped claim, not a universal one. It also cascades: with 3 fixtures in a directory, a golden failure on fixture #1 silently prevents fixtures #2 and #3 from ever running (confirmed empirically) — a suite of 12 cases would report nothing about 11 of them.
- **[HIGH]** The rubric judge's verdict extraction is a harness-side bug that can flip a correct verdict in either direction, undisclosed alongside the (honestly-taught) LLM-judge noise.
  - *Evidence:* Line 113: `verdict=$(... | $EVAL_JUDGE_CMD ... | grep -m1 -oE 'PASS|FAIL' | head -1)` scans the judge's entire output for the first line containing either keyword, even though the judge prompt (line 112) asks for 'ONE word on the first line.' I proved both failure directions with stubbed judges that reason before concluding: a genuinely bad (scope-bundling) answer was graded PASS because the judge's reasoning said '...tempting to PASS...' before its real 'Final verdict: FAIL'; and a genuinely good answer was graded FAIL because the judge's reasoning mentioned 'FAIL' while explaining why it did NOT fail, before its real 'Overall verdict: PASS'. This is additive noise on top of the disclosed 6pp infra-swing noise, and it is fixable (anchor to line 1, or require a fixed delimiter) but currently undisclosed.
- **[MEDIUM]** harness-metrics.sh's one load-bearing computed number — audit-check count — is measurably inflated by commented-out example code, on the kit's own reference template.
  - *Evidence:* Line 91's grep pattern `(pass|warn|fail)[[:space:]]+"` matches the shape of a call anywhere in a line's text regardless of a leading '#'. I counted this directly against claude-audit-base.sh: 18 of 75 matches (24%) sit inside comment-only lines (the INVARIANTS section's worked examples, lines 132-194), which never execute. A metric meant to answer 'is my safety net accreting checks' is contaminated by documentation from the very template it measures, and the contamination doesn't shrink predictably — it depends on how much of the template's illustrative commentary a user leaves in place.
- **[MEDIUM]** harness-metrics.sh omits eval-fixture count, an equally 'free to compute' number its sibling script already knows how to derive.
  - *Evidence:* claude-audit-base.sh lines 667-674 compute `n_evals=$(find "$ROOT/evals" -name '*.eval.md' ... | wc -l)`. harness-metrics.sh never references evals or eval-count anywhere (confirmed via grep, zero hits), despite item B and item A being introduced together in ROADMAP.md's build order and eval-count being at least as derivable as CLAUDE.md line count.
- **[MEDIUM]** The 7 'human note required' manual metrics have no scaffolding for how to compute them, and the trend log's schema is hardcoded to exactly two fields, so promoting a manual metric into the quantitative trend requires hand-editing the script's printf/delta calls.
  - *Evidence:* Lines 106-112 stub review-rounds, guard-fire frequency, %-merged-without-rework, defects-by-humans-vs-tests, escaped defects, rollbacks, and effort-per-merged-change with zero pointer to a source or formula. The append line (145) is hardcoded: `printf '%s\tclaude_md_lines=%s\taudit_checks=%s\n'` — no extension point. The script's own comment (line 24, 'wire it in and promote it into the free-to-compute block above') assumes the bash literacy to do that, which cuts against the kit's stated non-expert teaching intention.
- **[MEDIUM]** The roadmap's own 'built + proven on fixtures' rigor, applied to items H, O, and X, was not applied to item A — and the specific untested path (a golden failure) is exactly where the bug lives.
  - *Evidence:* ROADMAP.md's entries for H ('Proven on real fixtures: anchor present → runs; renamed → WARN...'), O ('proven on a fixture matrix'), and X carry explicit verification annotations; the entry for A carries no such annotation despite the eval runner and templates existing and being referenced throughout kickoff/audit. This is consistent with what I found: the golden PASS path plainly was exercised (it works correctly) while the golden FAIL path plainly was not (it crashes on line 101 every time).
- **[LOW]** The eval fixture's approval_class field uses a vocabulary disjoint from the kit's own action-risk taxonomy, with no cross-reference and no worked example distinguishing the values.
  - *Evidence:* evals-template/README.md line 46: `approval_class: read-only  # read-only | local-write | external`. claude-project-kickoff.md §1.3c defines a completely separate taxonomy (reversibility × reach → low/medium/high tier → auto/ask/deny/dry-run/batch-cap gate) for the same underlying concept ('how risky is this action'), and neither doc mentions the other. Both shipped example fixtures use the same value ('read-only'), so a non-expert has no worked example for when to pick 'local-write' or 'external', nor guidance on what it would ever gate.
- **[LOW]** Neither shipped example fixture demonstrates a non-code workflow, despite the ROADMAP explicitly framing this item as extending 'beyond code' to any checkable output (analyst reports, ops runbooks).
  - *Evidence:* cite-changed-file.eval.md is a unified-diff/file-path task; scope-check.eval.md is a README-typo/copyright-year scenario — both repo-flavored. The parsing mechanics themselves are genuinely code-agnostic (plain markdown, grep/sed only, EVAL_CMD is any text-in/text-out command), so the mechanical claim holds, but a non-expert bootstrapping a first eval for, say, a citation-checked analyst summary has nothing to pattern-match against.

## Fit against the kit's three intentions

### As a kickoff point for new projects

Good in intent, undermined by an unverified failure path. A new project copies claude-eval-base.sh verbatim to scripts/eval.sh, and the tiering advice (throwaway needs none; a maintained project seeds a few now, grows to 8-15) is honest and matches the kit's tiering elsewhere. But the very first time a real regression trips a golden fixture — the exact moment the tool exists to catch — it crashes with a cryptic bash internals error instead of reporting FAIL, and kickoff §1.6b only tells the adopter to run the happy path ('exit 0 = every eval held'), never to deliberately break something and confirm the FAIL path works. harness-metrics.sh fares better for kickoff specifically, since a brand-new project has no audit.sh yet to pollute the audit-check count.

### As an adoption skill for existing projects

Mixed. harness-metrics.sh's degrade-gracefully design (skip a missing CLAUDE.md or audit.sh rather than crash) is a real strength for a half-adopted, in-progress retrofit. But an adopting project inherits the identical crash bug the moment it seeds evals/ and hits a genuine regression — same failure as kickoff. Worse for adoption specifically: a mature project's real, hand-accumulated scripts/audit.sh will have its own comments and worked-example remnants, so the audit-check contamination I measured (24% on the reference template) isn't a one-time seeding artifact that fades — it can persist indefinitely and undermines exactly the long-horizon trend-tracking this script exists for in a mature repo.

### As a teaching tool beyond vibecoding

This is where the honest noise disclosure earns real credit — a newcomer taking the next step beyond vibecoding gets an unusually candid, well-sourced education in why an LLM judge isn't ground truth, which is precisely the kind of calibrated skepticism that distinguishes an informed builder from someone who just trusts whatever the agent says. But the crash-on-regression bug is close to the worst possible experience for that exact audience: someone new to bash and new to evals who sees 'scripts/eval.sh: line 101: expected�: unbound variable' gets zero teaching signal from it — it reads as their own mistake or a broken tool, not 'your eval just caught something.' Likewise, the manual-metrics section of harness-metrics.sh hands a newcomer a checklist of things to count with no worked example of how, which is a missed teaching moment exactly where claude-audit-base.sh (five worked TODO examples for a comparable job) shows the kit knows how to do this well elsewhere.

## Understand deeply vs. trust Claude

A builder must deeply own: (1) the golden-vs-rubric choice itself — this is judgment the tool cannot make for you, and defaulting to rubric where a golden answer would do reintroduces exactly the judge noise the kit warns about; (2) durable skepticism toward any LLM-judge verdict, including this harness's own rubric grading — the demonstrated verdict-extraction bug means a builder who never reads the judge's raw output when a rubric result looks surprising is exposed to a real, proven false-positive/false-negative path, not just the disclosed model noise; (3) that a clean eval run only proves the tested paths were tested — this investigation's central finding is that the golden PASS path worked while the golden FAIL path, by all evidence, was never actually run before shipping, and only a builder who deliberately breaks something and re-runs would ever learn which claim they actually have. What is safe to trust Claude to have written well here: the fixture FORMAT (frontmatter + H2 sections) and the happy-path plumbing, both of which I verified work correctly end to end; the sourced, consistent noise-caveat prose; and the defensive numeric-trend arithmetic in harness-metrics.sh (the 10# guard, the non-fabrication discipline).

## Durability over the next 1–2 years

What stays vital, reasoned from the roadmap's own §4 claim that 'reliably evaluating non-deterministic agents' is the field's hardest, most durable problem: the NEED for a judgment-layer verifier (evals) and a way to see whether the harness is paying off (a scorecard) does not shrink as models improve, because 'did this change help or regress the agent' is a comparison problem that scales with capability rather than one that better models solve outright — a stronger model still needs a saved judgment to check against. The provenance rule ('a naked factual claim is a defect') is similarly durable — it is a discipline about the shape of trustworthy output, not a patch for a weak model. What is likely to be replaced or become unnecessary: the specific mechanics — a bespoke grep/sed-only fixture parser (a concession to 'assume no jq/python3' that a more mature ecosystem would replace with a real eval framework, the way pytest subsumed hand-rolled test runners), the raw `claude -p`-and-hope invocation seam, and possibly the sharp golden/rubric split itself as judge calibration improves and the ~6pp infra-swing noise shrinks — though the roadmap is right to treat that last one skeptically rather than assume it resolves soon. The 8-15-fixture sizing guidance will likely look small within 1-2 years as agent autonomy and task horizon grow (the roadmap's own citation of METR's ~7-month time-horizon doubling implies eval suites need to grow at a comparable pace, not stay static).

## Missing beyond the roadmap (A–Y)

- No mechanism or reminder to actually exercise the eval runner's own FAIL path (or the scorecard's numeric edge cases) before trusting it — the kit preaches 'prove it bites, don't trust a self-report' everywhere else (H's safeguard-rot check, O's conformance verifier) but never turns that discipline on this specific artifact, which is exactly why the golden-FAIL crash shipped undetected.
- No cost/budget translation for the eval suite itself: the docs repeatedly say evals 'cost tokens and shell out to a live model' but never give a number a non-expert could budget against (e.g., fixture count × model calls × current pricing), unlike item B's own text which cites a concrete '>20x cost' figure for a comparable harness-cost claim.
- No reproducibility discipline for the rubric judge's own identity/version — a stored rubric verdict has no record of which judge model graded it, so a result from months ago isn't comparable to today's if the judge model was swapped or upgraded, and nothing here extends item J's 're-verify after a Claude Code upgrade' habit to 're-verify after a judge-model change.'
- No pointer connecting a caught eval regression to where its story should live (HARNESS_LOG.md entry? a wiki incident page? both?) — the kit is careful about this kind of routing everywhere else (the routing rule: guardrail → CLAUDE.md, machine-check → script, full story → wiki) but the evals layer stops at 'investigate before shipping' with no destination named.

## Exceptional

- The LLM-judge noise disclosure: specific (a named 6-percentage-point infra-swing figure), consistently repeated verbatim across three-plus docs, and it actually shapes the tool's own design choices (prefer golden, keep rubric blunt) rather than sitting as a disclaimer bolted on after the fact.
- harness-metrics.sh's refusal to fabricate unmeasured data: 'human note required' printed literally in place of a number, for 7 of 9 tracked dimensions, rather than a silent or fake 0 — genuinely rare restraint for a scorecard tool, where the far more common failure mode is presenting an incomplete count as if it were complete.
